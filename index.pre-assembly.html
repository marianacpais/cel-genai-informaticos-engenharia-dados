04
O SQL que mente · 1/4

"Quantos clientes temos?"

NL → Quantos clientes temos?
⟶ o que o LLM gera 
SELECT COUNT(*)
FROM encomendas;

→ 14 820 (encomendas)

⟶ o que estava certo 
SELECT COUNT(DISTINCT cliente_id)
FROM clientes;

→ 3 102 (clientes)

Pitfall: grain. A tabela tem uma linha por encomenda. Um cliente com 5 compras conta 5×. O LLM não conhece o grão da tua tabela.

05
O SQL que mente · 2/4

"Tempo médio de resolução de tickets"

NL → Qual o tempo médio de resolução de tickets?
⟶ o que o LLM gera 
SELECT AVG(fechado_em - aberto_em)
FROM tickets;

→ 1.8 dias (parece ótimo)

⟶ a realidade 
-- tickets ABERTOS (fechado_em IS NULL)
-- nem entram na média

→ muitos há semanas em aberto

Pitfall: linhas em falta. Os tickets ainda abertos não têm data de fecho → caem fora. A média parece melhor do que a realidade.

06
O SQL que mente · 3/4

"Produtos que nunca foram vendidos"

NL → Que produtos nunca foram vendidos?
⟶ o que o LLM gera 
SELECT * FROM produtos
WHERE id NOT IN (
  SELECT produto_id FROM vendas);

→ 0 linhas (sem erro!)

⟶ o que estava certo 
SELECT * FROM produtos p
WHERE NOT EXISTS (
  SELECT 1 FROM vendas v
  WHERE v.produto_id = p.id);

→ 37 produtos

Pitfall: NOT IN + NULL. Basta um produto_id a NULL na subquery para NOT IN devolver conjunto vazio. Footgun clássico de SQL.

07
O SQL que mente · 4/4

"Receita total por região"

NL → Qual a receita total por região?
⟶ o que o LLM gera 
SELECT regiao, SUM(valor)
FROM encomendas e
JOIN linhas_encomenda l
  ON l.encomenda_id = e.id
GROUP BY regiao;

→ receita inflada

⟶ o que estava certo 
-- agregar ao grão certo PRIMEIRO,
-- depois juntar à região
-- (senão o valor multiplica
--  pelo nº de linhas da encomenda)

→ receita real

Pitfall: fan-out do join. Várias linhas por encomenda → o valor é contado em dobro (ou triplo). O join "parece" certo.

08
O SQL que mente · síntese

Porque é que isto é pior do que um bug

O SQL é sintaticamente perfeito e corre sem erro.

O erro é semântico e silencioso — um número errado, não um crash.

Propaga-se a jusante durante semanas: corrupção silenciosa de dados.

O LLM não conhece o teu grão, as tuas definições de negócio, nem os teus NULLs.

o copiloto acelera quem sabe ler SQL · é armadilha para quem o usa para não aprender

Passo 2 · Matriz de embeddings

Um vetor por token — mas estático

Existe uma tabela vocabulário × d (ex.: 50 000 × 768). Cada token tem a sua linha — o vetor inicial. Ir buscá-lo é um lookup, não uma conta.

O problema: banco tem um só vetor, dê no que der. É o nível word2vec, de 2013.

Disfarçado de IA

O que faz um RAG falhar

Não é o modelo nem o prompt. É dados rançosos, schema drift, backfills inconsistentes e ausência de contratos — failure modes de dados.

Transversal

O fio crítico

Fio crítico · 1

Estocástico vs determinístico

Engenharia de dados

Determinística

Idempotente, reprodutível, auditável. A mesma entrada dá sempre a mesma saída.

LLM

Estocástico

Corre 2× → saídas diferentes. Mete-lo no meio do pipeline parte a garantia.

30
Fio crítico · 2

O custo de uma alucinação em dados

Não é um crash. É um número errado que propaga a jusante durante semanas.

corrupção silenciosa · invisível · pior do que um bug

Quando NÃO usar um LLM

Se uma regra determinística resolve (parsing de logs, formatos fixos), é mais barata, mais rápida e auditável. O LLM é para onde as regras não chegam.
