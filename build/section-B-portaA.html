O copiloto do engenheiro de dados

A porta esperada. O LLM senta-se ao lado de quem já faz o trabalho e acelera-o. Vamos atravessá-la com firmeza — mas com olho crítico, porque é aqui que mora o erro mais perigoso: o que corre sem dar erro. Estrutura: primeiro desmontamos o mito do "não é preciso saber SQL" com onze armadilhas reais; depois mostramos onde a GenAI acelera mesmo o engenheiro.
Porta A · o mito

"Já não é preciso escrever SQL — os LLMs escrevem-no"

Vamos provar, por A + B, que isto é perigosamente falso.

Onze perguntas em português. Onze SQLs lindos. Onze números errados — e nenhum crash.

A tese desta secção

O LLM não conhece o teu grão, as tuas definições de negócio, nem o comportamento dos teus NULLs. Sabe SQL; não sabe a tua base de dados.

NN
O perigo não é o SQL que rebenta — esse vê-se. É o SQL sintaticamente perfeito que devolve um número plausível e errado. Vou mostrar onze armadilhas, espalhadas por vendas, suporte, RH, saúde e logística. Todas universais. O domínio só muda a roupagem.
As 11 armadilhas · 1/11 · grain

"Quantos clientes temos?"

NL → Quantos clientes temos?
⟶ o que o LLM gera 
SELECT COUNT(*)
FROM encomendas;

→ 14 820 (encomendas)

⟶ o que estava certo 
SELECT COUNT(DISTINCT cliente_id)
FROM clientes;

→ 3 102 (clientes)

Pitfall: grain. A tabela tem uma linha por encomenda. Um cliente com 5 compras conta 5×. O LLM não conhece o grão da tua tabela.

NN
"Grão" é o que uma linha representa. Aqui a pergunta é sobre clientes mas a tabela está ao grão da encomenda. 4,7× de erro. Quem não sabe ler SQL nunca apanha isto.
As 11 armadilhas · 2/11 · linhas em falta

"Tempo médio até à primeira consulta"

NL → Qual o tempo médio de espera até à primeira consulta?
⟶ o que o LLM gera 
SELECT AVG(data_consulta - data_referencia)
FROM referencias
WHERE data_consulta IS NOT NULL;

→ 18 dias (parece excelente)

⟶ a realidade 
-- 1 240 referências ainda SEM
-- consulta marcada nem entram
-- na média. A fila real é outra.

→ mediana de espera 142 dias

Pitfall: linhas em falta. Quem ainda espera não tem data_consulta → cai fora do denominador. A média mede só quem já foi atendido. A minha armadilha preferida.

NN
A mesma armadilha veste-se de "tempo de resolução de tickets" no suporte. O viés de sobrevivência: medimos os que sobreviveram ao processo. Os que ainda esperam — os que mais importam — são invisíveis para a média. O número diz 18 dias; a realidade clínica são meses.
As 11 armadilhas · 3/11 · NOT IN + NULL

"Armazéns que nunca expediram nada"

NL → Que armazéns nunca expediram uma encomenda?
⟶ o que o LLM gera 
SELECT * FROM armazens
WHERE id NOT IN (
  SELECT armazem_id FROM expedicoes);

→ 0 linhas (sem erro!)

⟶ o que estava certo 
SELECT * FROM armazens a
WHERE NOT EXISTS (
  SELECT 1 FROM expedicoes e
  WHERE e.armazem_id = a.id);

→ 3 armazéns

Pitfall: NOT IN + NULL. Basta uma expedicoes.armazem_id a NULL para NOT IN devolver conjunto vazio. Lógica de três valores: x NOT IN (…, NULL) nunca é verdadeiro.

NN
Footgun clássico de SQL. O NULL na subquery contamina toda a comparação NOT IN. NOT EXISTS é imune porque trabalha por correspondência, não por pertença. O LLM escolhe NOT IN porque é a forma mais "lida" no treino — não porque é segura.
As 11 armadilhas · 4/11 · definição ambígua

"Qual o produto mais vendido?"

NL → Qual é o produto mais vendido?
⟶ o LLM escolhe um critério em silêncio 
SELECT produto, COUNT(*) n
FROM linhas_venda
GROUP BY produto
ORDER BY n DESC LIMIT 1;

→ Caneta (por nº de linhas)

⟶ "mais vendido" não é uma métrica 
-- por unidades:  SUM(qtd)
--   → Monitor 27"
-- por receita:   SUM(qtd*preco)
--   → Portátil

→ três produtos diferentes

Pitfall: definição ambígua. "Mais vendido" pode ser nº de transações, unidades ou receita. O LLM resolve a ambiguidade por ti — e não te diz qual escolheu.

NN
Diferente da armadilha 5: aqui não falta contexto de negócio escondido numa coluna, falta desambiguar a própria pergunta. A caneta ganha em nº de linhas, o portátil em receita. Nenhuma resposta está "errada" — está errada a confiança com que o LLM serve uma só.
As 11 armadilhas · 5/11 · definição de negócio

"Mostra-me os utilizadores ativos"

NL → Quantos utilizadores ativos temos?
⟶ o LLM agarra a coluna óbvia 
SELECT COUNT(*)
FROM utilizadores
WHERE estado = 'ativo';

→ 8 410 (a coluna até existe)

⟶ "ativo" no negócio = a usar o produto 
SELECT COUNT(DISTINCT utilizador_id)
FROM sessoes
WHERE inicio > NOW() - INTERVAL '30 days';

→ 2 933

Pitfall: definição de negócio. estado='ativo' significa "conta não suspensa", não "utilizador a usar". O LLM apanha a palavra, não o conceito. 2,9× de diferença.

NN
A armadilha mais traiçoeira porque a coluna existe e o nome bate certo. "Ativo" é um termo de negócio com definição própria — utilizador ativo mensal — que vive na cabeça da equipa, não no schema. O LLM não tem acesso a essa convenção. Aqui é onde uma camada semântica / métricas definidas faz toda a diferença.
As 11 armadilhas · 6/11 · fan-out do join

"Valor faturado por médico"

NL → Qual o valor total faturado por médico?
⟶ o que o LLM gera 
SELECT m.nome, SUM(c.valor)
FROM consultas c
JOIN atos a ON a.consulta_id = c.id
JOIN medicos m ON m.id = c.medico_id
GROUP BY m.nome;

→ valor inflado (+210%)

⟶ o que estava certo 
-- somar c.valor UMA vez por consulta.
-- O join a `atos` repete a linha da
-- consulta por cada ato → c.valor
-- entra 3× numa consulta com 3 atos.

→ valor real

Pitfall: fan-out do join. Juntar uma tabela de grão mais fino multiplica as linhas do lado "1". O SUM conta o mesmo valor várias vezes. O join "parece" certo.

NN
Primo da armadilha do grão, mas agora introduzido pelo join, não pela tabela base. Regra de ouro: agregar ao grão certo ANTES de juntar, ou somar com SUM(DISTINCT) só quando faz sentido. A faturação por médico fica 3× maior — e o erro escala com o nº médio de atos por consulta, por isso varia por especialidade. Quase impossível de apanhar a olho no resultado.
As 11 armadilhas · 7/11 · NULL no GROUP BY

"Salário médio por departamento"

NL → Qual o salário médio por departamento?
⟶ o que o LLM gera 
SELECT departamento, AVG(salario)
FROM colaboradores
GROUP BY departamento;

→ tabela "completa" (7 linhas)

⟶ o que o número esconde 
-- 47 colaboradores com
-- departamento = NULL formam um
-- grupo "fantasma" fácil de saltar.
-- E part-time entra sem normalizar.

→ a média global não bate

Pitfall: NULL no GROUP BY. Linhas com a chave de agrupamento a NULL juntam-se num grupo só, discreto e fácil de ignorar. O total parcial não reconcilia com o total geral.

NN
Dois problemas num: o grupo NULL silencioso e a falta de normalização (part-time/estágios a entrar na média sem ajuste de FTE). O teste rápido: a soma das partes bate com o todo? Se não, há um grupo a fugir — quase sempre NULL. O LLM nunca pergunta "e quem não tem departamento?".
As 11 armadilhas · 8/11 · unidades e limiar

"Doentes com glicemia alta"

NL → Quantos doentes têm glicemia alta?
⟶ o LLM crava um limiar 
SELECT COUNT(*)
FROM analises
WHERE glicemia > 126;

→ 5 110 (um número qualquer)

⟶ o contexto que falta 
-- mg/dL e mmol/L na MESMA coluna:
--   7,0 mmol/L = 126 mg/dL
-- "alta" depende de jejum vs ocasional
-- (126 em jejum ≠ 126 pós-prandial)

→ a contagem não tem sentido

Pitfall: unidades e limiar. O 126 é um número de manual aplicado sem saber a unidade nem a condição da colheita. Mistura mmol/L com mg/dL e ainda assume jejum. Plausível, preciso e errado.

NN
O LLM sabe que 126 mg/dL é o limiar de diabetes em jejum — e despeja-o. Não sabe que esta coluna mistura unidades, nem que metade das colheitas é ocasional. Em saúde isto é perigoso; em qualquer domínio com unidades (€ vs cêntimos, kg vs g) é o mesmo erro. O limiar tem de vir do negócio, não do treino do modelo.
As 11 armadilhas · 9/11 · soft delete

"Quantas encomendas este mês?"

NL → Quantas encomendas tivemos este mês?
⟶ o que o LLM gera 
SELECT COUNT(*)
FROM encomendas
WHERE data >= date_trunc('month', now());

→ 1 980

⟶ o que estava certo 
SELECT COUNT(*)
FROM encomendas
WHERE data >= date_trunc('month', now())
  AND cancelada = false;

→ 1 642

Pitfall: soft delete. As encomendas canceladas não são apagadas — ficam com uma flag. Sem o filtro cancelada = false, contam como reais. 17% a mais.

NN
Soft delete é a regra em sistemas modernos: nada se apaga de verdade, marca-se. O LLM não sabe que existe uma coluna `cancelada` (ou `deleted_at`) a não ser que o schema lho diga — e mesmo com o schema, raramente a aplica sozinho. Toda a métrica deste tipo de tabela precisa do filtro. É a convenção mais esquecida.
As 11 armadilhas · 10/11 · semântica de datas

"Entregas em 2025"

NL → Quantas entregas houve em 2025?
⟶ o LLM escolhe uma data qualquer 
SELECT COUNT(*)
FROM entregas
WHERE YEAR(data_envio) = 2025;

→ 22 410 (por envio)

⟶ qual data conta? 
-- "entrega em 2025" = entregue em 2025.
WHERE YEAR(data_entrega) = 2025
-- uma expedida 30/dez/24, entregue
-- 02/jan/25 muda de ano consoante a coluna.

→ 21 730 (por entrega)

Pitfall: semântica de datas. A mesma linha tem várias datas — envio, entrega, criação. O LLM agarra a primeira que parece encaixar. Fuso horário e cortes à meia-noite agravam.

NN
Qualquer evento tem múltiplos carimbos temporais e a pergunta raramente diz qual. "Vendas de dezembro" — por data de encomenda, de faturação, de pagamento? Some-se o fuso horário: um evento às 23h de Lisboa pode cair no dia seguinte em UTC. A diferença aqui é pequena no agregado, mas em relatórios de fecho de período é material.
As 11 armadilhas · 11/11 · empates no top-N

"Os 3 melhores vendedores do trimestre"

NL → Quem são os 3 melhores vendedores do trimestre?
⟶ o que o LLM gera 
SELECT vendedor, SUM(valor) total
FROM vendas
GROUP BY vendedor
ORDER BY total DESC
LIMIT 3;

→ corta o 4º que empata com o 3º

⟶ o que estava certo 
-- 3º e 4º empatados a 48 000 €.
-- LIMIT 3 escolhe UM ao acaso.
SELECT *, DENSE_RANK() OVER (
  ORDER BY total DESC) r
-- ... WHERE r <= 3   (decide a regra)

→ 4 vendedores (empate incluído)

Pitfall: empates no top-N. LIMIT não tem critério de desempate — devolve uma ordem arbitrária entre iguais. Hoje exclui o João, amanhã a Rita. Não-determinístico e injusto.

NN
Em RH isto é um problema de equidade: dois vendedores com o mesmo número, um fica de fora do bónus por acaso da ordenação. RANK/DENSE_RANK tornam a regra explícita e estável. O LLM gera o LIMIT porque é o padrão — não pensa em empates porque os dados de exemplo raramente os têm. É preciso pensar nos casos-limite, e isso é trabalho humano.

A defesa não é deixar de usar o copiloto; é saber ler o que ele produz. Liga à Aula 1: escrever → orientar → validar. A validação é onde mora o valor agora.

Porta A · porque o NL2SQL falha

O passo crítico: ligar a pergunta ao schema

O SQL é a parte fácil. O difícil é decidir que tabelas e colunas a pergunta quer dizer. É aqui que o NL2SQL erra — e onde tu, que conheces o modelo, és insubstituível.

NN
A literatura (NL2SQL, VLDB 2025) é clara: a gramática SQL os modelos já dominam; o gargalo é o schema linking — ligar termos da pergunta às entidades certas do schema. Numa BD com 300 tabelas e nomes ambíguos, é onde tudo descarrila. Dar ao modelo um bom dicionário de dados, uma camada semântica e exemplos melhora muito mais do que trocar de modelo. O conhecimento do domínio é a tua vantagem.

Porta A · criticar um modelo dbt gerado

O modelo "compila" — e está errado

dbt → receita_diaria.sql, gerado pelo copiloto
⟶ o que o LLM gera 
-- models/receita_diaria.sql
SELECT
  e.data::date AS dia,
  SUM(l.valor) AS receita
FROM {{ ref('encomendas') }} e
JOIN {{ ref('linhas') }} l
  ON l.enc_id = e.id
GROUP BY 1
-- (sem schema.yml, sem testes)

compila ✓ · receita inflada

⟶ a revisão de engenharia 
{{ config(materialized='incremental') }}
-- agregar linhas ANTES do join
-- (senão fan-out duplica receita)

-- schema.yml:
columns:
  - name: dia
    tests: [unique, not_null]
  - name: receita
    tests: [not_null]

testado ✓ · grão correto

O LLM gera o esqueleto, não o contrato. Faltam testes, falta o incremental, e o join repete a armadilha do fan-out. "Compila" não é "está certo".

NN
Repare como a armadilha 6 (fan-out) reaparece dentro de um modelo dbt — o copiloto não aprendeu a lição entre slides. A revisão acrescenta as três coisas que fazem um modelo de produção: o teste de unicidade no grão (dia), o not_null, e a materialização incremental para não reprocessar tudo. Isto é exatamente o code review que vale mais, não menos.
Porta A · migração de dialetos SQL

Traduzir Oracle → BigQuery não é trocar funções

Oracle → SELECT * FROM (… ORDER BY b DESC) WHERE ROWNUM <= 10;
⟶ tradução linha-a-linha do LLM 
-- BigQuery
SELECT nome, IFNULL(bonus,0) AS b
FROM emp
QUALIFY ROW_NUMBER() OVER () <= 10
ORDER BY b DESC;

→ 10 nomes ao acaso (corre!)

⟶ tradução fiel à semântica 
-- BigQuery
SELECT nome, IFNULL(bonus,0) AS b
FROM emp
ORDER BY b DESC
LIMIT 10;

→ os 10 maiores bónus

Em Oracle, ROWNUM filtra ANTES do ORDER BY. Daí o SELECT … FROM (subquery ordenada). O LLM traduz IFNULL bem, mas mantém a ordem das operações errada. Dialeto é semântica, não vocabulário.

NN
A migração é uma das melhores aplicações do copiloto — tarefa fechada, alto volume, padrões repetitivos. Mas tem caudas longas: a ordem de avaliação de ROWNUM, o tratamento de NULL em ordenações, datas e fusos, NUMBER vs FLOAT, sintaxe de janelas. O LLM acerta o caso comum e falha o canto. Por isso a migração assistida precisa de testes de equivalência: correr ambas e comparar resultados, não confiar na tradução.

Porta A · qualidade de dados semântica

O erro que passa em todas as regras

doente: 0 anos · diagnóstico: menopausa · medicação: terapêutica hormonal

Validação por regras

idade ≥ 0 ✓   não-nulo ✓   domínio válido ✓

cada campo passa isolado. A linha é aceite.

Deteção com LLM

lê a linha como um todo e apanha a incoerência clínica:

⚑ idade × diagnósticomenopausa aos 0 anosrever

O contexto é a anomalia. Nenhuma regra de coluna apanha isto — só a relação entre campos. O LLM é bom a sinalizar o improvável; o humano confirma.

NN
A qualidade de dados clássica é por regras: tipos, intervalos, não-nulos, expressões regulares. Apanha o malformado, não o implausível. Um bebé com menopausa passa em todos os testes de coluna — cada valor é válido por si. O LLM lê a linha como uma frase e estranha a combinação. Não substitui as regras: complementa-as, levantando candidatos a revisão. É deteção de anomalias semântica, com humano no fim — porque o LLM também inventa falsos positivos.
Porta A · lineage automático

Ler o código e desenhar o grafo de linhagem

O LLM lê as queries e infere "quem alimenta quem". Responde a "se eu mudar esta coluna, o que parte a jusante?" — sem ninguém ter desenhado o mapa.

NN
Lineage — linhagem — é o grafo de dependências entre tabelas e modelos. Ferramentas como o dbt já o derivam de `ref()`; o valor do LLM é fazê-lo a partir de SQL solto, procedimentos legados e DAGs do Airflow onde não há `ref()`. Responde à pergunta de ouro de impact analysis: mudo esta coluna, o que rebenta? Aviso: o grafo inferido pode falhar SQL dinâmico ou construído por strings — é um rascunho a validar, não a fonte de verdade.

Em 2026 o code review vale mais, não menos. Escrever SQL que corre deixou de impressionar. Ler SQL e apanhar a armadilha semântica — o grão, o NULL, a definição de negócio — é o que distingue.
