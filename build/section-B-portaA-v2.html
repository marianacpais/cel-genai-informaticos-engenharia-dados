Primeira abordagem

A GenAI ao serviço do engenheiro de dados

A primeira das duas abordagens: o LLM senta-se ao lado de quem já faz o trabalho e acelera-o. Atravessamos com olho crítico, porque é aqui que mora o erro mais perigoso — o que corre sem dar erro. Primeiro o mito do "não é preciso saber SQL", com alguns exemplos reais; depois onde a GenAI acelera de facto.
O mito

"Já não é preciso escrever SQL — os LLMs escrevem-no"

Os LLMs escrevem SQL impecável. Conhecer a sintaxe não é conhecer a base de dados.

Perguntas simples em português. SQL bonito. Números errados — e nenhum erro de execução.

A ideia central desta secção

O LLM não conhece o seu grão, as suas definições de negócio, nem o comportamento dos seus NULLs. Sabe SQL; não conhece a sua base de dados. Alguns exemplos — o resto explora-se no sandbox.

NN
O perigo não é o SQL que rebenta — esse vê-se. É o SQL sintaticamente perfeito que devolve um número plausível e errado. Aqui mostramos uma seleção; o sandbox "O SQL que mente" tem o conjunto completo, para experimentarem com o próprio chatbot.
O SQL que mente · linhas em falta

"Tempo médio até à primeira consulta"

NL → Qual o tempo médio de espera até à primeira consulta?
⟶ o que o LLM gera 
SELECT AVG(data_consulta - data_referencia)
FROM referencias
WHERE data_consulta IS NOT NULL;

→ 18 dias (parece excelente)

⟶ a realidade 
-- 1 240 referências ainda SEM
-- consulta marcada nem entram
-- na média. A fila real é outra.

→ mediana de espera 142 dias

Pitfall: linhas em falta. Quem ainda espera não tem data_consulta → cai fora do denominador. A média mede só quem já foi atendido — um viés de sobrevivência que torna o número irreconhecível.

NN
A mesma armadilha veste-se de "tempo de resolução de tickets" no suporte. O viés de sobrevivência: medimos os que sobreviveram ao processo. Os que ainda esperam — os que mais importam — são invisíveis para a média. O número diz 18 dias; a realidade são meses.
O SQL que mente · NOT IN + NULL

"Armazéns que nunca expediram nada"

NL → Que armazéns nunca expediram uma encomenda?
⟶ o que o LLM gera 
SELECT * FROM armazens
WHERE id NOT IN (
  SELECT armazem_id FROM expedicoes);

→ 0 linhas (sem erro)

⟶ o que estava certo 
SELECT * FROM armazens a
WHERE NOT EXISTS (
  SELECT 1 FROM expedicoes e
  WHERE e.armazem_id = a.id);

→ 3 armazéns

Pitfall: NOT IN + NULL. Basta uma expedicoes.armazem_id a NULL para NOT IN devolver conjunto vazio. Lógica de três valores: x NOT IN (…, NULL) nunca é verdadeiro. NOT EXISTS é imune.

NN
Footgun clássico de SQL. O NULL na subquery contamina toda a comparação NOT IN. NOT EXISTS é imune porque trabalha por correspondência, não por pertença. O LLM escolhe NOT IN porque é a forma mais frequente no treino — não porque é segura.
O SQL que mente · definição ambígua

"Qual o produto mais vendido?"

NL → Qual é o produto mais vendido?
⟶ o LLM escolhe um critério em silêncio 
SELECT produto, COUNT(*) n
FROM linhas_venda
GROUP BY produto
ORDER BY n DESC LIMIT 1;

→ Caneta (por nº de linhas)

⟶ "mais vendido" não é uma métrica 
-- por unidades:  SUM(qtd)
--   → Monitor 27"
-- por receita:   SUM(qtd*preco)
--   → Portátil

→ três produtos diferentes

Pitfall: definição ambígua. "Mais vendido" pode ser nº de transações, unidades ou receita. O LLM resolve a ambiguidade por si — e não diz qual critério escolheu.

NN
Aqui não falta contexto de negócio escondido numa coluna — falta desambiguar a própria pergunta. A caneta ganha em nº de linhas, o portátil em receita. Nenhuma resposta está "errada"; errada é a confiança com que o LLM serve uma só.
O SQL que mente · definição de negócio

"Mostra-me os utilizadores ativos"

NL → Quantos utilizadores ativos temos?
⟶ o LLM agarra a coluna óbvia 
SELECT COUNT(*)
FROM utilizadores
WHERE estado = 'ativo';

→ 8 410 (a coluna até existe)

⟶ "ativo" no negócio = a usar o produto 
SELECT COUNT(DISTINCT utilizador_id)
FROM sessoes
WHERE inicio > NOW() - INTERVAL '30 days';

→ 2 933

Pitfall: definição de negócio. estado='ativo' significa "conta não suspensa", não "utilizador a usar". O LLM apanha a palavra, não o conceito. 2,9× de diferença.

NN
A armadilha mais traiçoeira: a coluna existe e o nome bate certo. "Ativo" é um termo de negócio com definição própria — utilizador ativo mensal — que vive na convenção da equipa, não no schema. É onde uma camada semântica faz toda a diferença.
O SQL que mente · unidades e limiar

"Doentes com glicemia alta"

NL → Quantos doentes têm glicemia alta?
⟶ o LLM crava um limiar 
SELECT COUNT(*)
FROM analises
WHERE glicemia > 126;

→ 5 110 (um número qualquer)

⟶ o contexto que falta 
-- mg/dL e mmol/L na MESMA coluna:
--   7,0 mmol/L = 126 mg/dL
-- "alta" depende de jejum vs ocasional
-- (126 em jejum ≠ 126 pós-prandial)

→ a contagem não tem sentido

Pitfall: unidades e limiar. O 126 é um número de manual aplicado sem saber a unidade nem a condição da colheita. Mistura mmol/L com mg/dL e ainda assume jejum. Plausível, preciso e errado.

NN
O LLM sabe que 126 mg/dL é o limiar de diabetes em jejum — e despeja-o. Não sabe que esta coluna mistura unidades, nem que metade das colheitas é ocasional. Em qualquer domínio com unidades (€ vs cêntimos, kg vs g) é o mesmo erro. O limiar tem de vir do negócio, não do treino do modelo.
O SQL que mente · soft delete

"Quantas encomendas este mês?"

NL → Quantas encomendas tivemos este mês?
⟶ o que o LLM gera 
SELECT COUNT(*)
FROM encomendas
WHERE data >= date_trunc('month', now());

→ 1 980

⟶ o que estava certo 
SELECT COUNT(*)
FROM encomendas
WHERE data >= date_trunc('month', now())
  AND cancelada = false;

→ 1 642

Pitfall: soft delete. As encomendas canceladas não são apagadas — ficam com uma flag. Sem o filtro cancelada = false, contam como reais. 17% a mais.

NN
Soft delete é a regra em sistemas modernos: nada se apaga, marca-se. O LLM não sabe que existe uma coluna `cancelada` (ou `deleted_at`) a não ser que o schema lho diga — e mesmo com o schema, raramente a aplica sozinho. Toda a métrica deste tipo de tabela precisa do filtro. É a convenção mais esquecida.
O SQL que mente · empates no top-N

"Os 3 melhores vendedores do trimestre"

NL → Quem são os 3 melhores vendedores do trimestre?
⟶ o que o LLM gera 
SELECT vendedor, SUM(valor) total
FROM vendas
GROUP BY vendedor
ORDER BY total DESC
LIMIT 3;

→ corta o 4º que empata com o 3º

⟶ o que estava certo 
-- 3º e 4º empatados a 48 000 €.
-- LIMIT 3 escolhe UM ao acaso.
SELECT *, DENSE_RANK() OVER (
  ORDER BY total DESC) r
-- ... WHERE r <= 3   (decide a regra)

→ 4 vendedores (empate incluído)

Pitfall: empates no top-N. LIMIT não tem critério de desempate — devolve uma ordem arbitrária entre iguais. Hoje exclui um, amanhã outro. Não-determinístico e injusto.

NN
Em RH é um problema de equidade: dois vendedores com o mesmo número, um fica de fora do bónus por acaso da ordenação. RANK/DENSE_RANK tornam a regra explícita e estável. O LLM gera o LIMIT porque é o padrão — não pensa em empates porque os dados de exemplo raramente os têm. Pensar nos casos-limite é trabalho humano.

A GenAI ao serviço do engenheiro · porque o NL2SQL falha

O passo crítico: ligar a pergunta ao schema

O SQL é a parte fácil. O difícil é decidir que tabelas e colunas a pergunta quer dizer. É aqui que o NL2SQL erra — e onde quem conhece o modelo de dados é insubstituível.

NN
A literatura (NL2SQL, VLDB 2025) é clara: a gramática SQL os modelos já dominam; o gargalo é o schema linking — ligar termos da pergunta às entidades certas. Numa BD com 300 tabelas e nomes ambíguos é onde tudo descarrila. Dar ao modelo um bom dicionário de dados, uma camada semântica e exemplos rende mais do que trocar de modelo. O conhecimento do domínio é a vantagem da equipa.

A GenAI ao serviço do engenheiro · rever o modelo gerado

O modelo "compila" — e está errado

gerado → receita_diaria.sql, produzido pelo copiloto
⟶ o que o LLM gera 
SELECT
  e.data::date AS dia,
  SUM(l.valor) AS receita
FROM encomendas e
JOIN linhas l
  ON l.enc_id = e.id
GROUP BY 1
-- (sem testes)

compila ✓ · receita inflada

⟶ a revisão de engenharia 
-- agregar linhas ANTES do join
-- (senão o fan-out duplica a receita)
-- + testes:
--   dia    → único, não-nulo
--   receita → não-nulo
-- + materialização incremental

testado ✓ · grão correto

O LLM gera o esqueleto, não o contrato. Faltam os testes, e o join repete a armadilha do fan-out. "Compila" não é "está certo".

NN
[PENDENTE DE DECISÃO.] A armadilha do fan-out reaparece dentro de um modelo gerado — o copiloto não aprende a lição entre slides. A revisão acrescenta o que faz um modelo de produção: o teste de unicidade no grão, o não-nulo, e a materialização incremental. É exatamente o code review que vale mais, não menos.
A GenAI ao serviço do engenheiro · migração de dialetos

Traduzir Oracle → BigQuery não é trocar funções

Oracle → SELECT * FROM (… ORDER BY b DESC) WHERE ROWNUM <= 10;
⟶ tradução linha-a-linha do LLM 
-- BigQuery
SELECT nome, IFNULL(bonus,0) AS b
FROM emp
QUALIFY ROW_NUMBER() OVER () <= 10
ORDER BY b DESC;

→ 10 nomes ao acaso (corre)

⟶ tradução fiel à semântica 
-- BigQuery
SELECT nome, IFNULL(bonus,0) AS b
FROM emp
ORDER BY b DESC
LIMIT 10;

→ os 10 maiores bónus

Em Oracle, ROWNUM filtra ANTES do ORDER BY. O LLM traduz IFNULL bem, mas mantém a ordem das operações errada. Outras caudas: NULL em ordenações, datas e fusos, NUMBER vs FLOAT. Dialeto é semântica, não vocabulário.

NN
A migração é uma das melhores aplicações do copiloto — tarefa fechada, alto volume, padrões repetitivos. Mas tem caudas longas: a ordem de avaliação de ROWNUM, NULL em ordenações, datas e fusos, NUMBER vs FLOAT, sintaxe de janelas. O LLM acerta o caso comum e falha o canto. Por isso a migração assistida precisa de testes de equivalência: correr ambas e comparar resultados.

A GenAI ao serviço do engenheiro · qualidade semântica

O erro que passa em todas as regras

doente: 0 anos · diagnóstico: menopausa · medicação: terapêutica hormonal
admissão: 2026-05-10 · alta: 2026-05-03 (alta antes da admissão)

Validação por regras

idade ≥ 0 ✓   não-nulo ✓   datas válidas ✓

cada campo passa isolado. As linhas são aceites.

Deteção com LLM

lê a linha como um todo e apanha a incoerência:

⚑ menopausa aos 0 anos⚑ alta antes da admissãorever

O contexto é a anomalia. Nenhuma regra de coluna apanha isto — só a relação entre campos. O LLM é bom a sinalizar o improvável; o humano confirma.

NN
A qualidade clássica é por regras: tipos, intervalos, não-nulos. Apanha o malformado, não o implausível. Um bebé com menopausa, ou uma alta antes da admissão, passa em todos os testes de coluna — cada valor é válido por si. O LLM lê a linha como uma frase e estranha a combinação. Não substitui as regras: complementa-as, levantando candidatos a revisão (e inventa falsos positivos, por isso humano no fim).

A limitação comum a todos

Têm acesso ao schema, mas não à camada de negócio: o que é "ativo", que datas contam, onde está o soft delete. Sem essa camada, repetem as mesmas armadilhas dentro da plataforma.

NN
Não é preciso adotar nada novo — o copiloto já vem na plataforma que usam. Têm acesso ao schema, o que ajuda no schema linking, mas continuam sem a camada de negócio. A observação é frequente na comunidade de dados (ver SOURCES). A defesa de fundo é a mesma dos exemplos: uma camada semântica e um humano que sabe ler o resultado.
