Porta B

Construir para a GenAI

Viramos a página. Até aqui a GenAI foi copiloto — escrevia SQL, sugeria, e nós validávamos. Agora a relação inverte-se: somos nós, engenheiros de dados, a construir os pipelines de que a GenAI precisa para funcionar. Duas peças nucleares: o LLM como passo de transformação (ETL) e os embeddings. Começamos pelo ETL porque é o mais imediato — e o que melhor mostra que isto é engenharia, com método.
Porta B · LLM-como-ETL

O T do ETL pode ser um LLM

"Homem, 54 anos, HTA,
sem alergias conhecidas,
medicado com ramipril."
→
LLM+ schema (Pydantic)
→
{
  idade: 54,
  sexo: "M",
  diagnosticos: ["hipertensão"],
  alergias: [],
  medicacao: ["ramipril"]
}

No ETL clássico, o T são regras determinísticas: regex, CAST, JOIN, dicionários. Agora pode ser um modelo que lê texto livre e devolve um registo estruturado.

NN
O ETL tem três letras; a do meio é a que muda. Transformação era código que nós escrevíamos linha a linha. Aqui o input é prosa que nenhum parser apanha — a nota de triagem de um enfermeiro — e o output é um JSON com schema. Reparem em dois detalhes que já anunciam o resto da secção: "sem alergias conhecidas" tem de virar lista vazia, não a presença de alergias (negação); e "HTA" vira "hipertensão" (normalização de vocabulário). O modelo faz isto bem na maioria das vezes — e é o "maioria das vezes" que nos vai ocupar.

Exemplo 2 · suporte

Inferir o que está implícito — sem inventar o que falta

"Desde a atualização de ontem o vosso app crasha ao exportar PDF no Windows 11. Já reinstalei. Urgente!"
→
LLM+ schema
→
{
  categoria: "bug/export",
  severidade: "alta",
  plataforma: "Windows 11",
  passos_tentados: ["reinstalar"],
  pedido_reembolso: null
}

"Urgente" e "crasha" → severidade inferida. Reembolso nunca foi mencionado → null, não false inventado.

NN
Aqui há duas coisas a separar. A primeira é boa: o modelo infere severidade que não está escrita como campo — junta "urgente", "crasha", "desde a atualização" e classifica "alta". Isto é exatamente o que um regex nunca faria. A segunda é a armadilha: o cliente não diz nada sobre reembolso. A tentação do modelo é preencher `false`. Mas `false` é uma afirmação — "o cliente não quer reembolso" — e isso é falso; o cliente simplesmente não falou. A diferença entre `false` e `null` é a diferença entre uma resposta e uma alucinação. Voltamos a isto na falha das alucinações.
Exemplo 3 · jurídico

Números escondidos na prosa — e fáceis de trocar

cláusula → "Renova-se por períodos de 12 meses salvo denúncia com 60 dias de antecedência."
o erro plausível 
{
  "renovacao_automatica": true,
  "periodo_meses": 60,
  "pre_aviso_dias": 12
}

os dois números trocados

o certo 
{
  "renovacao_automatica": true,
  "periodo_meses": 12,
  "pre_aviso_dias": 60
}

período 12 · pré-aviso 60

Pitfall: dois inteiros na mesma frase, ambos plausíveis em ambos os campos. O schema garante o formato, não a semântica — por isso é que validamos contra regras de negócio (pré-aviso < período).

NN
Extração numérica a partir de prosa é onde os modelos escorregam de forma mais traiçoeira, porque o resultado parece válido. 60 meses de renovação e 12 dias de pré-aviso é um contrato perfeitamente concebível — só não é este. O schema-constraint não vos salva aqui: ambos são inteiros, ambos cabem nos dois campos. A defesa é validação de domínio: uma regra de negócio que diz "o pré-aviso é tipicamente menor que o período" levanta a flag para revisão humana. Isto é engenharia de dados clássica — constraints — aplicada à saída de um modelo.
Exemplo 4 · recibo (OCR)

Texto sujo, e a vírgula decimal portuguesa

OCR → MERCADO XPTO  TOTAL 23,90 EUR  IVA 6%  2026-06-12
o erro plausível 
{
  "total": 2390,
  "moeda": "EUR",
  "iva_pct": 6,
  "data": "12/06/2026"
}

total 2390 · data ambígua

o certo 
{
  "total": 23.90,
  "moeda": "EUR",
  "iva_pct": 6,
  "data": "2026-06-12"
}

total 23.90 · data ISO

Pitfall: a vírgula decimal PT (23,90) vira 2390 se o modelo assumir convenção anglo-saxónica. Localização não é detalhe — é correção.

NN
Dois problemas portugueses num só recibo. A vírgula decimal: 23,90 são vinte e três euros e noventa cêntimos, não dois mil trezentos e noventa. Um modelo treinado maioritariamente em inglês trata a vírgula como separador de milhares e multiplica a fatura por cem. E a data: "2026-06-12" já vem em ISO, mas o modelo pode "normalizar" para dd/mm e introduzir ambiguidade onde não havia. A lição: a localização — moeda, decimais, datas — é um requisito de correção, e deve estar explícita no prompt e validada à saída (`total` é um decimal positivo plausível para um recibo de mercearia).
Exemplo 5 · CV

Sintetizar factos que o texto não nomeia

"5 anos como backend dev em Python e Go; liderou equipa de 4; mestrado em EI pela FEUP."
→
LLM+ schema
→
{
  anos_experiencia: 5,
  linguagens: ["Python","Go"],
  gestao: true,
  grau: "Mestrado",
  instituicao: "FEUP"
}

"Liderou equipa" → gestao: true — um campo que nenhuma palavra do texto contém. E siglas resolvidas: EI, FEUP.

NN
Este é o exemplo que mostra a força genuína do LLM. O campo `gestao: true` não corresponde a nenhuma palavra — não existe "gestão" no texto. O modelo lê "liderou equipa de 4" e sintetiza um booleano. Um regex não tem hipótese; precisaríamos de uma lista interminável de padrões ("liderou", "geriu", "coordenou", "foi responsável por..."). E as siglas: "EI" é Engenharia Informática, "FEUP" é a faculdade do Porto — conhecimento de mundo que o modelo traz. Mas atenção: esta mesma capacidade de sintetizar é a porta da alucinação. A fronteira entre "inferir gestão" e "inventar reembolso" é ténue, e é por isso que precisamos de schema e validação.
Exemplo 6 · morada

Parsing de morada, com campos que podem faltar

"R. das Flores 12 3ºEsq,
4000-001 Porto"
→
LLM+ schema
→
{
  rua: "Rua das Flores",
  numero: "12",
  andar: "3º Esq",
  codigo_postal: "4000-001",
  cidade: "Porto"
}

Expande R. → Rua, separa andar do número, valida o formato do CP. O andar é opcional — numa morada sem ele, fica null.

NN
Moradas são o exemplo perfeito de campos opcionais. Esta tem andar; a próxima pode não ter. Um schema rígido que exigisse "andar" partia metade dos dados. Por isso o campo é `Optional` e a regra é "null se ausente", não "inventa um 1º andar". O modelo também faz trabalho útil de normalização: abrevia "R." para "Rua", reconhece o formato de código postal português 4000-001. Reparem que isto é parcialmente regex-able — o CP tem um padrão fixo — e por isso uma abordagem híbrida (regex para o CP, LLM para o resto) é muitas vezes a melhor. Liga ao "quando não usar LLM".
Exemplo 7 · review

Sentimento não é um número — é por aspeto

"Adorei o material mas veio com um risco na lateral, e a entrega atrasou 3 dias."
→
LLM+ schema
→
{
  sentimento: "misto",
  aspetos: [
   {aspeto:"material", pol:"+"},
   {aspeto:"acabamento", pol:"−"},
   {aspeto:"entrega", pol:"−"}
  ],
  problema_entrega: true
}

Um rótulo único ("3 estrelas?") deita fora a informação. Por aspeto, o produto é elogiado e a logística é o problema — duas equipas diferentes.

NN
A análise de sentimento ingénua devolve um número — "negativo", ou três estrelas. Mas esta review não é simplesmente negativa: o cliente adorou o produto e detestou a entrega. Colapsar isso num escalar perde exatamente a informação acionável. A extração por aspeto separa material (positivo), acabamento (negativo) e entrega (negativo) — e de repente a tabela diz à empresa que o problema é a logística, não o produto. Reparem que o modelo até inferiu um aspeto "acabamento" a partir de "risco na lateral", que não está nomeado. Esta estrutura — lista de objetos com aspeto e polaridade — é onde o schema-constraint brilha: força a forma, e o modelo preenche.
Exemplo 8 · linha de log

O caso em que o regex bate o LLM

log → 2026-06-12T14:03:11Z ERROR db-pool timeout after 30000ms host=pg-primary

LLM

lê, "compreende", devolve JSON — e gasta uma chamada de API, ~300ms e cêntimos por linha. Não-determinístico. A milhões de linhas/dia, insustentável.

Regex / parser

o formato é fixo e estável. Uma expressão apanha tudo.

determinístico~microssegundoscusto zeroauditável

Regra: texto semi-estruturado e estável → código determinístico. O LLM aqui é um martelo a matar uma mosca — e a mosca foge de cada vez de forma diferente.

NN
Pus este exemplo no meio dos dez de propósito, para vos tirar do encantamento. Uma linha de log tem formato fixo: timestamp ISO, nível, componente, mensagem, host. Isto é um caso de escola para um regex — uma expressão captura todos os grupos, corre em microssegundos, é determinística e auditável, e custa zero. Meter um LLM aqui é absurdo em três eixos: custo (chamada de API por linha, a milhões de linhas), latência (300ms vs microssegundos) e, o pior, não-determinismo — a mesma linha pode dar saídas ligeiramente diferentes, o que num pipeline de observabilidade é veneno. Guardem esta intuição: o LLM paga-se quando o texto é genuinamente livre. Quando há estrutura, há ferramenta melhor.
Exemplo 9 · carta de alta

Documento longo: o que cabe, e mantém-se consistente?

Carta de alta — 3 páginas
história, exames, evolução,
medicação, plano de seguimento…
→
LLMcontexto longo
→
{
  diagnostico_principal: …,
  diagnosticos_secundarios: […],
  medicacao_alta: […],
  seguimento: …
}
Os dois riscos do texto longo

Cabe no contexto? três páginas cabem; um processo de 200 cabe mas dilui a atenção. E consistência: a medicação citada na história tem de bater certo com a da alta — o modelo pode contradizer-se ao longo do documento.

NN
Documentos longos trazem problemas que os exemplos curtos não têm. Primeiro, o contexto: estas três páginas cabem, mas um processo clínico de 200 páginas, mesmo cabendo na janela, faz a atenção do modelo diluir — o que está a meio tende a ser pior recuperado (o efeito "lost in the middle"). Segundo, a consistência interna: a medicação mencionada na história clínica tem de coincidir com a lista de medicação à alta, e o modelo pode extrair versões diferentes de secções diferentes. Estratégias de engenharia: dividir o documento por secções e extrair de cada uma, depois reconciliar; ou pedir citações (de que parte do texto veio cada campo) para auditar. Isto liga diretamente ao chunking que vimos nos embeddings.
Exemplo 10 · anúncio imobiliário

Vários tipos numa só frase curta

"T2 remodelado, 78m², 2º andar com elevador, 320.000€, Cedofeita"
→
LLM+ schema
→
{
  tipologia: "T2",
  area_m2: 78,
  andar: 2,
  elevador: true,
  preco: 320000,
  zona: "Cedofeita"
}

Numa frase: enum (T2), inteiros (78, 2), booleano (elevador), preço sem separador (320000), texto livre (zona).

NN
O anúncio imobiliário é o oposto da carta de alta: tudo numa frase, mas com cinco tipos de dados diferentes encavalitados. "T2" é um enum (tipologia portuguesa), "78m²" precisa de descartar a unidade e ficar com o inteiro, "2º andar" extrai o número e separa-o da tipologia, "com elevador" vira um booleano, "320.000€" tem de perder o ponto de milhares e o símbolo, "Cedofeita" é texto livre. Um regex daria imenso trabalho para um formato que varia de anúncio para anúncio. Aqui o LLM justifica-se: variabilidade alta, frase curta, custo controlado. É o contraponto exato do log.
Falha 1 · negação

"Sem X" tem de virar ausência, não presença

texto → "… sem alergias conhecidas …"
o que falha 
{
  "alergias": ["alergias"]
}

apanhou a palavra, perdeu o "sem"

o correto 
{
  "alergias": []
}

a negação virou ausência

Porquê importa: em saúde, "sem alergias" registado como "tem alergias" é um erro clínico. A negação é semântica, não lexical — e é onde a extração ingénua mais tropeça.

NN
A negação é a falha número um porque é silenciosa e perigosa. Uma extração que apanha a palavra "alergias" e ignora o "sem" inverte o significado clínico — regista que o doente tem alergias quando não tem. Modelos modernos lidam bem com negações simples, mas complicam-se com negações de âmbito longo ("nega febre, tosse ou dispneia nas últimas 48h" — quantos campos isso negar?) e com dupla negação. A defesa não é só confiar: é testar explicitamente casos de negação no golden set e, em domínios críticos, validar. Reparem que uso azeitona para o erro, nunca vermelho — é a regra da identidade CeL.
Falha 2 · normalização

Uma só forma canónica — mas quem a define?

A pergunta que a IA não responde

Qual é a forma canónica? "hipertensão"? O código ICD-10 I10? O conceito SNOMED? Esse vocabulário é uma decisão de governança de dados — tua, não do modelo.

NN
"HTA", "tensão alta", "hipertensão arterial" e "hipertensão" são a mesma coisa para um humano, e o modelo até as colapsa razoavelmente. Mas a pergunta difícil não é se o modelo normaliza — é normalizar para quê. Qual é a forma canónica? Se diferentes pipelines escolherem formas diferentes ("hipertensão" aqui, código I10 ali), os teus dados não se juntam. A normalização de vocabulário é um problema de governança de dados tão antigo quanto os data warehouses — terminologias controladas, ICD, SNOMED, master data. O LLM é uma ferramenta poderosa para mapear texto livre para o teu vocabulário, mas o vocabulário tens de o definir tu. O modelo executa a política; não a decide.
Falha 3 · alucinação

Preencher o que o texto não diz

e-mail → … nada é dito sobre reembolso …
o que falha 
{
  "pedido_reembolso": false
}

afirma algo que o texto não diz

o correto 
{
  "pedido_reembolso": null
}

distingue ausente de falso

Mitigação: instrução explícita — "devolve null se o campo não estiver presente no texto" — e tipos Optional no schema. O modelo tende a preencher; tem de ser travado.

NN
A alucinação de campos é a sombra da capacidade de inferir. O mesmo modelo que corretamente deduz "gestao: true" do CV vai, se o deixarmos, deduzir "pedido_reembolso: false" de um e-mail que nunca falou em reembolso. A diferença é que no CV há evidência ("liderou equipa") e aqui não há nenhuma. `false` é uma afirmação positiva de ausência de pedido; `null` é honestidade sobre o que o texto não cobre. Para downstream isto é enorme: um relatório que conta "clientes que não querem reembolso" baseado em `false` alucinados é lixo. A mitigação tem duas pernas: a instrução explícita "null se ausente" no prompt, e tipos `Optional` no schema que permitem o null em vez de forçar um default.
Falha 4 · não-determinismo

A mesma entrada, duas saídas

execução 1 
{
  "severidade": "alta",
  "categoria": "bug/export"
}
execução 2 · mesmo input 
{
  "severidade": "média",
  "categoria": "exportação"
}
A colisão com a idempotência

A engenharia de dados exige que correr o pipeline 2× dê o mesmo resultado. Um LLM no T parte essa garantia. Mitigar: temperature=0, fixar versão do modelo, e cache por hash do input — re-extrair só se o texto mudar.

NN
Esta é a falha que mais incomoda um engenheiro de dados, porque ataca um princípio, não um valor. Idempotência: correr o mesmo pipeline sobre os mesmos dados tem de dar o mesmo output, senão não há reprodutibilidade, não há testes fiáveis, não há reconciliação. Um LLM é não-determinístico por natureza — até com temperatura zero há variação, e uma atualização silenciosa do modelo muda tudo. As mitigações: `temperature=0` reduz mas não elimina; fixar a versão exata do modelo nos metadados; e a mais importante para um pipeline — cache por hash do input, ou seja, guardas a extração e só voltas a chamar o modelo se o texto de origem mudar. Assim o pipeline volta a ser idempotente na prática, mesmo que o modelo não seja. Isto liga diretamente ao slide da idempotência no início da aula.
Porta B · o critério

Regra determinística ou LLM?

Regra determinística

formato fixo e estável · auditabilidade exigida · volume alto / latência baixa · custo previsível.

logs, CSV, código postal, datas ISO, IDs

LLM justifica-se

linguagem livre e variável · inferência ou síntese semântica · cauda longa de formatos imprevisíveis.

negaçãosíntesesentimentosiglas

Regra de ouro: se um regex resolve, não metas um LLM. Ele é caro, lento e não-determinístico — paga-se só quando o texto é genuinamente livre. E o híbrido (regex + LLM no que sobra) ganha muitas vezes.

NN
Recolhemos a intuição dos dez exemplos numa decisão. A coluna da esquerda — regra determinística — vence sempre que o formato é fixo e estável, sempre que a auditabilidade é legalmente exigida, sempre que o volume é massivo ou a latência crítica, e sempre que precisas de custo previsível. O log é o caso de escola. A coluna da direita — LLM — só se justifica quando o texto é genuinamente livre: variabilidade linguística alta, necessidade de inferir ou sintetizar, uma cauda longa de formatos que nenhum regex cobre. E reparem na nota de rodapé: a resposta certa muitas vezes não é "ou/ou" mas híbrido — regex apanha o código postal e o LLM trata do resto da morada. Engenharia é escolher a ferramenta mais barata que resolve, não a mais impressionante.
Porta B · a resposta de engenharia · 1/2

O schema é um contrato, não uma sugestão

structured output com Pydantic 
from pydantic import BaseModel, Field

class NotaClinica(BaseModel):
    idade: int = Field(ge=0, le=120)
    sexo: str                      # enum no prompt
    diagnosticos: list[str] = []
    alergias: list[str] = []       # [] quando "sem alergias"
    medicacao: list[str] = []
    confianca: float = Field(ge=0, le=1)

# o modelo é OBRIGADO a devolver este shape
resp = client.extract(texto, schema=NotaClinica)

JSON válido por construção — não por sorte

NN
Vocês são informáticos, por isso mostro o código. O schema-constraint — structured output, function calling, ou Pydantic com uma biblioteca como o Instructor — transforma "por favor devolve JSON" numa garantia. O modelo é forçado, ao nível da geração de tokens, a produzir algo que encaixa neste tipo. `idade` é um inteiro entre 0 e 120, não a string "cinquenta e quatro". `alergias` é uma lista, com default lista vazia, o que reforça a regra da negação. E acrescentei um campo `confianca`: peço ao próprio modelo uma estimativa de quão seguro está, que uso a seguir para flag de revisão. Sem schema-constraint, o JSON parte — chaves a mais, vírgulas a faltar, prosa à volta — e o teu parser estoira em produção. Com ele, o JSON é válido por construção.

Validação de domínio à saída — tipos, intervalos, e regras de negócio como a do contrato — para apanhar a semântica que o schema não apanha. E flag de baixa confiança, usando aquele campo `confianca`, para mandar o duvidoso para um humano em vez de o deixar entrar silenciosamente nos dados. A mensagem do painel é a tese da Porta B inteira: a GenAI no pipeline não dispensa a disciplina de engenharia de dados — schema, validação, idempotência, linhagem — ela exige-a com mais força, porque acrescenta uma fonte de erro nova e não-determinística. Quem trata o LLM como uma caixa fechada constrói dívida; quem o trata como mais um passo de transformação com contrato constrói sistemas.

Porta B · embeddings · escolher e avaliar

Já sabes como se produzem. Agora, qual escolher?

Abrimos a caixa dos embeddings: tokenização, matriz, atenção, pooling, treino contrastivo, e as consequências de engenharia. Tudo isso era o "como funciona". Esta segunda parte é o "como escolher e avaliar" — as decisões práticas que tomas quando montas um sistema real: que família de modelo, que dimensão, modelo geral ou de domínio, e o português, e como medir se está a funcionar. Menos teoria, mais decisão de arquitetura.
Tipos de modelo · 1

De uma palavra-um-vetor a uma frase-um-vetor

word2vec / GloVe 2013 · estático · 1 vetor por palavra, sem contexto
BERT e descendentes 2018 · contextual · o vetor depende da frase
sentence-transformers 2019+ · frase → 1 vetor, pooling treinado para semelhança

O que pões num vector store é quase sempre da terceira camada: um sentence-transformer, BERT afinado com pooling para que a semelhança cosseno entre frases signifique algo.

NN
Três gerações. Word2vec e GloVe, de 2013, são estáticos: cada palavra tem um vetor fixo, "banco" é sempre o mesmo independentemente da frase — vimos isto no core. BERT, 2018, trouxe o contextual: o vetor do token depende da frase à volta. Mas BERT cru dá um vetor por token e não é ótimo para comparar frases inteiras. Os sentence-transformers, de 2019 em diante, pegam num BERT e afinam-no especificamente para que o vetor da frase, depois do pooling, viva num espaço onde cosseno alto significa "querem dizer o mesmo". Esta terceira camada é o que usas em RAG. Quando alguém diz "modelo de embedding" hoje, quase sempre quer dizer um sentence-transformer. A distinção seguinte — bi-encoder vs cross-encoder — é dentro desta família.
Tipos de modelo · 2

Bi-encoder para procurar, cross-encoder para reordenar

Bi-encoder

codifica a query e o documento em separado — um vetor cada. Pré-calculas os vetores dos docs e indexas. Rápido, escala a milhões.

é o que vai para o vector store · retrieval

Cross-encoder

mete query e documento juntos no modelo e devolve uma pontuação de par. Muito mais preciso — e não indexável.

precisolentorerank do top-k

Padrão real: bi-encoder recupera os top-50 (barato), cross-encoder reordena esses 50 (caro mas só 50). O melhor dos dois.

NN
Esta distinção é decisiva para quem constrói RAG. O bi-encoder codifica a pergunta e cada documento separadamente — duas passagens independentes pelo modelo, um vetor cada. A grande vantagem: os vetores dos documentos pré-calculam-se e indexam-se, então no momento da query só codificas a pergunta e fazes busca vetorial. Escala a milhões. A desvantagem: como nunca viu a pergunta e o documento ao mesmo tempo, perde interações finas. O cross-encoder faz o oposto: mete os dois juntos no modelo e produz uma pontuação de relevância do par. Muito mais preciso, mas não dá um vetor reutilizável — terias de correr o modelo para cada par query-documento, impossível a milhões. Por isso o padrão de produção é em duas fases: o bi-encoder recupera um top-50 barato, o cross-encoder reordena só esses 50. Recall barato, precisão onde importa.
Dimensionalidade

Mais dimensões: mais nuance, mais custo

384 leve · índice barato · menos nuance
768 o equilíbrio mais comum
1536 / 3072 mais nuance · storage e índice HNSW caros

A dimensão multiplica tudo: storage do vector store, memória do índice, custo de cada comparação. A milhões de vetores, escolher 1536 quando 384 chegava é dinheiro a arder — decisão de arquitetura.

NN
As dimensões típicas: 384, 768, 1024, 1536, e os grandes vão a 3072. Mais dimensões dão ao modelo mais eixos para codificar nuance, e em geral mais qualidade de retrieval. Mas a dimensão multiplica todos os custos: o storage do vector store é linear na dimensão, a memória do índice HNSW também, e cada cálculo de semelhança percorre o vetor inteiro. A milhões de vetores, a diferença entre 384 e 1536 é quatro vezes o storage e o custo de índice — pode ser a diferença entre caber na RAM ou não. Por isso isto é uma decisão de arquitetura, não um default. A pergunta certa é: a tua tarefa precisa mesmo da nuance extra, ou um modelo de 384 dimensões recupera igualmente bem no teu corpus? E isto liga ao próximo conceito, que parte este trade-off pela raiz: e se pudesses escolher a dimensão depois de treinar?
Matryoshka (MRL)

Um vetor que se pode encurtar

Treinado para que os primeiros valores carreguem o essencial. Cortas o vetor em 256 e ainda funciona — uma só geração, várias dimensões. Indexas curto para triagem, guardas longo para rerank.

NN
Matryoshka Representation Learning, MRL, resolve o trade-off da dimensão de forma elegante — daí o nome das bonecas russas. O modelo é treinado com uma perda que obriga os primeiros 256 valores do vetor a já serem, sozinhos, um bom embedding; os primeiros 768 a serem melhores; e o vetor completo o melhor de todos. A consequência prática é poderosa: geras o vetor uma vez, em dimensão máxima, mas podes truncá-lo — literalmente cortar e renormalizar — para a dimensão que quiseres, sem re-embeber. Isto permite um padrão muito eficiente: indexas a versão curta de 256 para fazer triagem rápida e barata sobre milhões, e guardas a versão longa para reordenar com precisão só o top que sobrou. Os modelos de embedding mais recentes da OpenAI e vários open-source suportam isto. É a resposta de engenharia ao slide anterior: já não tens de escolher a dimensão antes de saber se precisas dela.

Lembra do core: "parecido" é definido pelos pares de treino. Se o teu domínio (clínico, jurídico, código) tem semântica própria, ou usas um modelo de domínio, ou afinas com os teus pares.

NN
No core vimos o corolário honesto do treino contrastivo: "parecido" é exatamente o que os pares de treino disserem que é parecido. Um modelo geral foi treinado em texto da internet, onde "Java" tanto é ilha como linguagem como café, e ele coloca-os todos a uma distância morna. Para um motor de busca de documentação técnica, isso é mau — quero que "Java" puxe "JVM" e empurre "ilha tropical". A solução é modelo de domínio: ou existe um pré-treinado para a tua área (há modelos clínicos, jurídicos, de código), ou afinas um modelo geral com pares teus — exemplos de "isto é parecido com aquilo no meu domínio". O fine-tuning de embeddings é dos mais acessíveis: alguns milhares de pares positivos do teu corpus chegam para mover o espaço. A decisão é de qualidade de dados: o modelo geral é fraco no teu jargão? Mede primeiro, antes de assumir que precisas de afinar.

A regra

O leaderboard é quase todo em inglês. Testa no teu PT, com o teu domínio — não extrapoles da pontuação em inglês.

NN
O português tem armadilhas próprias. Do lado bom: um modelo multilingual decente alinha as línguas no mesmo espaço, então "hipertensão" e "hypertension" ficam perto, e podes pesquisar em português sobre um corpus em inglês — útil. Do lado mau: muitos modelos foram treinados esmagadoramente em inglês e são medíocres em português; e quando há português no treino, costuma ser PT-BR, que diverge do PT-PT em vocabulário ("rato" vs "mouse", termos clínicos, jurídicos). Há também um custo escondido: o português gasta mais tokens que o inglês — acentos, contrações, morfologia rica — então o mesmo texto enche mais a janela e cabe menos por chunk, o que muda as tuas contas de chunking. E o jargão local — terminologia clínica ou jurídica portuguesa — raramente está bem representado. A conclusão prática: as tabelas de classificação de embeddings são quase todas em inglês; não extrapoles. Mede no teu português, com o teu domínio. O que nos leva à última peça: como medir.

É pouco trabalho — uma tarde — e é o ativo mais valioso que tens, porque te deixa medir qualquer modelo, qualquer estratégia de chunking, qualquer dimensão, no que realmente importa. Versiona-o como versionas código. O leaderboard escolhe os finalistas; o teu golden set escolhe o vencedor.

As métricas que dizem se o retrieval presta

Recall@ko documento certo está no top-k? A métrica-rei do RAG — se não recupera, o LLM não tem com que responder

MRRquão acima está o primeiro acerto (1/posição). Penaliza pôr o certo em 9º

nDCG@kordenação ponderada por relevância graduada, não só sim/não

Precision@kquanto do top-k é lixo? Trade-off com o recall

Para RAG, começa pelo recall@k

Sobe o k até o recall ser alto o suficiente; depois usa rerank (cross-encoder) para a precisão. Recuperar mal é falhar antes de o LLM abrir a boca.

NN
Quatro métricas, mas uma manda. Recall@k: dos documentos relevantes, quantos apanhaste no top-k? Para RAG é a métrica-rei, porque o que não é recuperado simplesmente não chega ao LLM — ele não pode responder com o que não vê. MRR, mean reciprocal rank: olha para a posição do primeiro acerto, recompensa pô-lo em primeiro e penaliza pô-lo em nono. nDCG: o mais sofisticado, pondera por relevância graduada e pela posição, para quando "relevante" não é binário. Precision@k: a fração do top-k que é mesmo relevante — importa quando enches o contexto do LLM com ruído. A estratégia prática para RAG: foca-te primeiro no recall@k, sobe o k até recuperares quase sempre o documento certo, e só depois trazes um cross-encoder para reordenar e melhorar a precisão no topo. A ordem importa: recuperar mal é falhar antes de o gerador sequer começar.
