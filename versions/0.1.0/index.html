Estágio 2 · Ingestão

Batch vs streaming

Quase tudo o que parece exigir "tempo real" vive bem com batch de poucos minutos. Streaming é uma decisão de custo, não de moda.

05
Batch: junta os dados e processa-os de tempos a tempos (de hora a hora, à noite). Simples de raciocinar, fácil de reprocessar, barato. Streaming: processa cada evento à medida que chega, latência de milissegundos a segundos — mas obriga a pensar em janelas, dados que chegam fora de ordem, estado, exactly-once. Heurística honesta: a maioria dos casos de "negócio em tempo real" tolera micro-batch de 1–5 min. Só vale streaming verdadeiro quando a latência é mesmo do produto (deteção de fraude, monitorização, recomendação ao vivo). Esta escolha vai voltar na Porta B: indexar embeddings para RAG é tipicamente batch/CDC; servir a query é online.
Estágio 3 · Armazenamento

Warehouse, lake, lakehouse

Data Warehouseestruturado · schema-on-write · SQL e BI
Data Lakebruto · schema-on-read · ficheiros, ML, qualquer formato
Lakehouseconvergência · tabelas ACID sobre o lake (Delta · Iceberg)

O warehouse exige estrutura à entrada; o lake aceita tudo e adia a estrutura. O lakehouse tenta o melhor dos dois — e é onde a base vetorial do RAG vai morar ao lado.

06
Warehouse (BigQuery, Snowflake, Redshift): impõe schema na escrita, otimizado para SQL analítico e BI; ótimo para tabelas, mau para PDFs e imagens. Data lake (S3, ADLS + Parquet): armazenamento barato que aceita qualquer ficheiro, schema só na leitura; flexível, mas vira facilmente "data swamp" sem governança. Lakehouse (Databricks/Delta, Iceberg): formatos de tabela transacionais por cima dos ficheiros do lake — transações ACID, time travel, schema enforcement — convergindo as duas tradições. Porque importa para esta aula: os dados não-estruturados (texto, que alimenta os embeddings) vivem no lake/lakehouse; a base de dados vetorial entra como um novo vizinho neste stack, como veremos na Porta B.
Estágio 4 · Transformação

ETL vs ELT — a ordem mudou

O armazém ficou barato e elástico → carrega-se cru e transforma-se lá dentro. O T passou a ser SQL versionado — e, na Porta B, pode até ser um LLM.

07
ETL clássico: transformava-se num servidor próprio antes de carregar, porque o armazenamento era caro e só queríamos lá dentro o dado limpo. ELT inverte: a cloud tornou o armazém barato e com computação elástica, por isso carrega-se o dado cru e transforma-se com SQL lá dentro — versionado, testado, com ferramentas como dbt. Vantagens do ELT: o dado cru fica disponível para reprocessar, a transformação é código revisível, escala com o armazém. Foreshadow deliberado: quando, na Porta B, dissermos "o T do ETL pode ser um LLM", é exatamente este passo de Transform que estamos a substituir/aumentar — e é por isso que herda todos os problemas de determinismo que veremos.
Transformação · modelação

O grão: o que uma linha representa

Modelar é decidir, para cada tabela, o que é uma linha. Esse é o grão (grain).

Errar o grão é a raiz de metade dos números errados: contagens infladas, médias diluídas, somas em dobro. E é precisamente o que um LLM não consegue adivinhar ao ler só o nome da tabela.

Guardem esta palavra

Na Porta A, o "SQL que mente" vai falhar quase sempre por grão. A modelação é a defesa.

encomendas
1 linha = 1 encomenda

um cliente com 5 compras → 5 linhas
COUNT(*) conta encomendas, não clientes

08
Grain = o nível de detalhe de uma linha de facto. "Uma linha por encomenda", "uma linha por linha-de-encomenda", "uma linha por cliente por dia" — são grãos diferentes e dão respostas diferentes à mesma pergunta. Boa modelação dimensional (estrela: factos no grão certo + dimensões) existe justamente para tornar as agregações inequívocas. Liga diretamente à Porta A: a demo do text-to-SQL falha por desconhecer o grão (contar encomendas em vez de clientes distintos, somar valor com fan-out de join). O LLM lê o schema, não a semântica — o grão é conhecimento de negócio que mora na cabeça da equipa, não no nome das colunas.

Transversal · orquestração

O maestro: Airflow, Dagster

DAGgrafo de tarefas com dependências, sem ciclos

Schedulingquando corre — @daily, cron, ou por evento

Retriesfalhou? tenta de novo com backoff, sem intervir

Porque é que isto exige idempotência

Se uma tarefa pode correr de novo (retry, backfill) sem o teu dedo, ela tem de poder repetir-se sem estragar. Segura este fio.

10
A orquestração é a camada que costura os cinco estágios: decide a ordem, dispara no tempo certo, reage a falhas. Airflow é o padrão histórico (DAGs em Python, scheduling rico, enorme ecossistema); Dagster é a geração mais nova, orientada a "assets" de dados e a testes/observabilidade desde a raiz. Conceitos a fixar: o DAG (Directed Acyclic Graph) — tarefas com dependências e sem ciclos, para que haja sempre uma ordem de execução; scheduling (tempo ou evento); retries com backoff exponencial; backfill (correr o pipeline para datas passadas). O ponto pedagógico crítico: retries e backfill só são seguros se as tarefas forem idempotentes — é a ponte para os próximos dois slides, e a propriedade que vai colidir com os LLMs.
Orquestração · um DAG

Um DAG, por dentro

Dois extratos correm em paralelo, juntam-se, passam pelo gate de qualidade — e só então carregam. O load_dw sobrescreve a partição: pode correr 10×, dá o mesmo.

11
Ler o grafo: dois extratos independentes (orders, users) podem correr ao mesmo tempo — o orquestrador explora o paralelismo permitido pelas dependências. Convergem num join (fan-in), passam por um quality_check que funciona como gate (se falhar, o load não corre — evita publicar dados maus), e só então load_dw. O retry ×3 com backoff em clean_orders mostra a tolerância a falhas transitórias (uma API que timeout). E repare-se no rótulo "overwrite partição" no load: é a escolha de design que torna a tarefa idempotente — correr de novo não duplica, sobrepõe. Acíclico é essencial: se houvesse um ciclo, não haveria ordem de execução possível. Tudo isto é o terreno onde, mais à frente, se discute pôr um LLM no meio — e porque é que isso parte a garantia.
A propriedade-chave · 1/2

Idempotência: correr 2× = correr 1×

A regra: desenhar tarefas que se podem repetir sem efeito colateral — sobrescrever uma partição, fazer upsert por chave. Não INSERT cego.

12
Idempotência (termo da matemática/REST): uma operação é idempotente se aplicá-la N vezes tem o mesmo efeito que aplicá-la uma vez. Em dados: reprocessar uma partição deve substituí-la, não acumular. À esquerda, overwrite da partição do dia — o retry/backfill é seguro. À direita, um INSERT cego: cada corrida acrescenta de novo as mesmas linhas, e o backfill duplica silenciosamente os factos (médias erradas, somas infladas). Técnicas para garantir: overwrite por partição, upsert/MERGE por chave natural, chaves determinísticas, marcar lotes já processados. Esta é a propriedade que dá fiabilidade à fábrica — e é exatamente a que um passo estocástico (um LLM) não respeita, como veremos a seguir.
A propriedade-chave · 2/2

Reprodutibilidade — e a colisão que aí vem

A mesma entrada, o mesmo código, a mesma versão → o mesmo resultado. Sempre. É isto que torna os dados auditáveis.

determinístico · versionado · re-executável

A colisão que vai assombrar a aula

Um LLM é estocástico: corre 2× → saídas diferentes. Metê-lo no meio do pipeline parte a idempotência e a reprodutibilidade de uma só vez. Guardem isto para a Porta B.

13
Reprodutibilidade é a irmã da idempotência: não só repetir não estraga, como repetir produz exatamente o mesmo output — o que exige versionar tudo (código, dados de entrada, dependências, configuração). É a base da auditabilidade: poder explicar, meses depois, porque é que um número era aquele. Conceitos relacionados: pinned versions, lockfiles, data snapshots, lineage. Aqui planta-se de propósito a tensão central do curso: a engenharia de dados foi construída sobre determinismo; os LLMs são estocásticos por natureza (temperatura, amostragem, não-determinismo até a temperatura 0 por questões de hardware/batching). Quando, na Porta B, colocarmos um LLM como passo de transformação, esta garantia parte-se — e a resposta de engenharia (schema-constrain, validação, caching, flags de confiança) é precisamente tentar reconstruí-la.

O que o contrato garante

O produtor não pode mudar o schema à socapa. Uma alteração que parta o contrato é uma breaking change — versionada, comunicada, testada no CI. Sem contrato, cada mudança a montante é uma surpresa a jusante.

14
Um data contract é um acordo explícito e versionado entre quem produz os dados (a equipa da app) e quem os consome (a equipa de dados): define schema, tipos, nullability, valores permitidos, SLAs de frescura e volume, e a semântica de cada campo (o que "estado=ativo" significa). O problema que resolve: na maioria das organizações o produtor desconhece quem consome a jusante, e uma renomeação inocente de coluna parte dashboards e modelos sem ninguém saber. Com contrato, o schema é uma interface pública — mudanças que o quebram passam por versionamento e validação em CI, como uma API. Ferramentas/ideias: enforcement no broker (schema registry para eventos), testes de contrato, expectations. Liga ao slide seguinte: a ausência ou violação de contrato manifesta-se como schema drift.
Fiabilidade · a falha

Schema drift: quando o contrato se quebra

O produtor mudou o tipo de um campo. O pipeline não rebenta logo — soma errado durante semanas. Corrupção silenciosa, outra vez.

15
Schema drift é a deriva, ao longo do tempo, da estrutura dos dados de origem: um campo muda de tipo (number → string), é renomeado, deixa de vir, passa a aceitar NULL, ou ganha um valor de enum novo. No exemplo, "amount" passou de número para string formatada ("12,50€") — talvez uma alteração inocente no frontend. O consumidor que faz SUM(amount) ou rebenta (melhor caso, falha visível) ou — pior — faz cast silencioso para NULL/0 e a soma fica errada sem ninguém ver, durante semanas. É o mesmo padrão de "corrupção silenciosa" do SQL que mente. Defesas: data contracts (slide anterior), schema enforcement no lakehouse, testes de contrato, e observabilidade (slide a seguir) para detetar a deriva cedo. Mensagem de transição: precisamos de instrumentar a fábrica para ver estas falhas, não esperar que um humano tropece nelas.

16
As dimensões clássicas de qualidade de dados (completude, validade, consistência, unicidade, atualidade — algumas frameworks acrescentam exatidão e integridade referencial). O ponto operacional: cada uma traduz-se em testes executáveis que correm no pipeline a cada corrida — "esta coluna não pode ter NULL", "este valor está no enum permitido", "a contagem de hoje não pode cair >20% face à média", "a chave é única". Ferramentas: dbt tests, Great Expectations, Soda. A diferença entre uma fábrica amadora e uma profissional é esta: a qualidade é asserida por código que falha o pipeline (ou dispara um gate, como o quality_check do DAG), em vez de descoberta por um analista zangado três semanas depois. Liga à Porta A: a GenAI pode propor testes de qualidade semânticos (erros contextuais de uma linha que regras fixas não apanham).

17
Observabilidade de dados é o equivalente, para dados, do APM/monitoring de software. Os cinco pilares habituais (popularizados pela Monte Carlo): freshness (chegou a horas?), volume (a contagem de linhas está no intervalo esperado?), distribuição/schema (os valores e a estrutura mantêm a forma?), e lineage (o grafo de dependências coluna-a-coluna, essencial para impact analysis: "se mudar esta fonte, o que parto a jusante?"). Por cima, deteção de anomalias: baselines aprendidas do histórico que disparam alertas sem limiares fixos. Distinção pedagógica importante face ao slide anterior: testes de qualidade verificam hipóteses que tu já formulaste (known unknowns); observabilidade existe para os unknown unknowns — a falha que ninguém previu, como o drift silencioso. 18
Governança de dados é o conjunto de políticas e processos que mantêm os dados fiáveis, seguros e usáveis à escala da organização: ownership, definições partilhadas, controlo de acesso, privacidade (PII, RGPD, retenção), e o catálogo de dados — o inventário pesquisável de todos os ativos, com descrições, donos e linhagem. Para esta turma de informáticos não-saúde, mantenho deliberadamente breve (o curso tem sessões dedicadas). O gancho que importa: a qualidade dos metadados é diretamente o teto de desempenho dos copilotos da Porta A — um LLM que escreve SQL é tão bom quanto a documentação que o alimenta. Catálogo rico → copiloto útil; catálogo pobre → confiança a fazer joins errados. É a ponte "lixo à entrada" do fio crítico do deck.

O LLM não conhece o teu grão, as tuas definições de negócio, nem o comportamento dos teus NULLs. Sabe SQL; não sabe a tua base de dados.

  WHERE e.armazem_id = a.id);

→ 3 armazéns

Pitfall: NOT IN + NULL. Basta uma expedicoes.armazem_id a NULL para NOT IN devolver conjunto vazio. Lógica de três valores: x NOT IN (…, NULL) nunca é verdadeiro.

24
Footgun clássico de SQL. O NULL na subquery contamina toda a comparação NOT IN. NOT EXISTS é imune porque trabalha por correspondência, não por pertença. O LLM escolhe NOT IN porque é a forma mais "lida" no treino — não porque é segura.

"Mostra-me os utilizadores ativos"

As 11 armadilhas · 6/11 · fan-out do join

"Valor faturado por médico"

NL → Qual o valor total faturado por médico?
⟶ o que o LLM gera 
SELECT m.nome, SUM(c.valor)
FROM consultas c
JOIN atos a ON a.consulta_id = c.id
JOIN medicos m ON m.id = c.medico_id
GROUP BY m.nome;

→ valor inflado (+210%)

⟶ o que estava certo 
-- somar c.valor UMA vez por consulta.
-- O join a `atos` repete a linha da
-- consulta por cada ato → c.valor
-- entra 3× numa consulta com 3 atos.

→ valor real

Pitfall: fan-out do join. Juntar uma tabela de grão mais fino multiplica as linhas do lado "1". O SUM conta o mesmo valor várias vezes. O join "parece" certo.

27
Primo da armadilha do grão, mas agora introduzido pelo join, não pela tabela base. Regra de ouro: agregar ao grão certo ANTES de juntar, ou somar com SUM(DISTINCT) só quando faz sentido. A faturação por médico fica 3× maior — e o erro escala com o nº médio de atos por consulta, por isso varia por especialidade. Quase impossível de apanhar a olho no resultado.

As 11 armadilhas · 9/11 · soft delete

"Quantas encomendas este mês?"

NL → Quantas encomendas tivemos este mês?
⟶ o que o LLM gera

FROM encomendas

→ 1 980

⟶ o que estava certo

Pitfall: soft delete. As encomendas canceladas não são apagadas — ficam com uma flag. Sem o filtro cancelada = false, contam como reais. 17% a mais.

Soft delete é a regra em sistemas modernos: nada se apaga de verdade, marca-se. O LLM não sabe que existe uma coluna `cancelada` (ou `deleted_at`) a não ser que o schema lho diga — e mesmo com o schema, raramente a aplica sozinho. Toda a métrica deste tipo de tabela precisa do filtro. É a convenção mais esquecida.

"Os 3 melhores vendedores do trimestre"

⟶ o que o LLM gera

FROM vendas

→ corta o 4º que empata com o 3º

⟶ o que estava certo 
-- 3º e 4º empatados a 48 000 €.

  ORDER BY total DESC) r
-- ... WHERE r <= 3   (decide a regra)

Porta A · migração de dialetos SQL

Traduzir Oracle → BigQuery não é trocar funções

Oracle → SELECT * FROM (… ORDER BY b DESC) WHERE ROWNUM <= 10;
⟶ tradução linha-a-linha do LLM 
-- BigQuery
SELECT nome, IFNULL(bonus,0) AS b
FROM emp
QUALIFY ROW_NUMBER() OVER () <= 10
ORDER BY b DESC;

→ 10 nomes ao acaso (corre!)

⟶ tradução fiel à semântica 
-- BigQuery
SELECT nome, IFNULL(bonus,0) AS b
FROM emp
ORDER BY b DESC
LIMIT 10;

→ os 10 maiores bónus

Em Oracle, ROWNUM filtra ANTES do ORDER BY. Daí o SELECT … FROM (subquery ordenada). O LLM traduz IFNULL bem, mas mantém a ordem das operações errada. Dialeto é semântica, não vocabulário.

38
A migração é uma das melhores aplicações do copiloto — tarefa fechada, alto volume, padrões repetitivos. Mas tem caudas longas: a ordem de avaliação de ROWNUM, o tratamento de NULL em ordenações, datas e fusos, NUMBER vs FLOAT, sintaxe de janelas. O LLM acerta o caso comum e falha o canto. Por isso a migração assistida precisa de testes de equivalência: correr ambas e comparar resultados, não confiar na tradução.

Exemplo 2 · suporte

Inferir o que está implícito — sem inventar o que falta

"Desde a atualização de ontem o vosso app crasha ao exportar PDF no Windows 11. Já reinstalei. Urgente!"
→

→

  categoria: "bug/export",
  severidade: "alta",
  plataforma: "Windows 11",
  passos_tentados: ["reinstalar"],
  pedido_reembolso: null
}

"Urgente" e "crasha" → severidade inferida. Reembolso nunca foi mencionado → null, não false inventado.

46
Aqui há duas coisas a separar. A primeira é boa: o modelo infere severidade que não está escrita como campo — junta "urgente", "crasha", "desde a atualização" e classifica "alta". Isto é exatamente o que um regex nunca faria. A segunda é a armadilha: o cliente não diz nada sobre reembolso. A tentação do modelo é preencher `false`. Mas `false` é uma afirmação — "o cliente não quer reembolso" — e isso é falso; o cliente simplesmente não falou. A diferença entre `false` e `null` é a diferença entre uma resposta e uma alucinação. Voltamos a isto na falha das alucinações.

Extração numérica a partir de prosa é onde os modelos escorregam de forma mais traiçoeira, porque o resultado parece válido. 60 meses de renovação e 12 dias de pré-aviso é um contrato perfeitamente concebível — só não é este. O schema-constraint não vos salva aqui: ambos são inteiros, ambos cabem nos dois campos. A defesa é validação de domínio: uma regra de negócio que diz "o pré-aviso é tipicamente menor que o período" levanta a flag para revisão humana. Isto é engenharia de dados clássica — constraints — aplicada à saída de um modelo.

Falha 2 · normalização

Uma só forma canónica — mas quem a define?

A pergunta que a IA não responde

A alucinação de campos é a sombra da capacidade de inferir. Falha 4 · não-determinismo

A mesma entrada, duas saídas

A colisão com a idempotência

A engenharia de dados exige que correr o pipeline 2× dê o mesmo resultado. Um LLM no T parte essa garantia. Mitigar: temperature=0, fixar versão do modelo, e cache por hash do input — re-extrair só se o texto mudar.

58
Esta é a falha que mais incomoda um engenheiro de dados, porque ataca um princípio, não um valor. Idempotência: correr o mesmo pipeline sobre os mesmos dados tem de dar o mesmo output, senão não há reprodutibilidade, não há testes fiáveis, não há reconciliação. Um LLM é não-determinístico por natureza — até com temperatura zero há variação, e uma atualização silenciosa do modelo muda tudo. As mitigações: `temperature=0` reduz mas não elimina; fixar a versão exata do modelo nos metadados; e a mais importante para um pipeline — cache por hash do input, ou seja, guardas a extração e só voltas a chamar o modelo se o texto de origem mudar. Assim o pipeline volta a ser idempotente na prática, mesmo que o modelo não seja. Isto liga diretamente ao slide da idempotência no início da aula.

59
Recolhemos a intuição dos dez exemplos numa decisão. A coluna da esquerda — regra determinística — vence sempre que o formato é fixo e estável, sempre que a auditabilidade é legalmente exigida, sempre que o volume é massivo ou a latência crítica, e sempre que precisas de custo previsível. O log é o caso de escola. A coluna da direita — LLM — só se justifica quando o texto é genuinamente livre: variabilidade linguística alta, necessidade de inferir ou sintetizar, uma cauda longa de formatos que nenhum regex cobre. E reparem na nota de rodapé: a resposta certa muitas vezes não é "ou/ou" mas híbrido — regex apanha o código postal e o LLM trata do resto da morada. Engenharia é escolher a ferramenta mais barata que resolve, não a mais impressionante.

Tipos de modelo · 1

De uma palavra-um-vetor a uma frase-um-vetor

word2vec / GloVe 2013 · estático · 1 vetor por palavra, sem contexto
BERT e descendentes 2018 · contextual · o vetor depende da frase
sentence-transformers 2019+ · frase → 1 vetor, pooling treinado para semelhança

O que pões num vector store é quase sempre da terceira camada: um sentence-transformer, BERT afinado com pooling para que a semelhança cosseno entre frases signifique algo.

70
Três gerações. Word2vec e GloVe, de 2013, são estáticos: cada palavra tem um vetor fixo, "banco" é sempre o mesmo independentemente da frase — vimos isto no core. BERT, 2018, trouxe o contextual: o vetor do token depende da frase à volta. Mas BERT cru dá um vetor por token e não é ótimo para comparar frases inteiras. Os sentence-transformers, de 2019 em diante, pegam num BERT e afinam-no especificamente para que o vetor da frase, depois do pooling, viva num espaço onde cosseno alto significa "querem dizer o mesmo". Esta terceira camada é o que usas em RAG. Quando alguém diz "modelo de embedding" hoje, quase sempre quer dizer um sentence-transformer. A distinção seguinte — bi-encoder vs cross-encoder — é dentro desta família.
Tipos de modelo · 2

Bi-encoder para procurar, cross-encoder para reordenar

Bi-encoder

codifica a query e o documento em separado — um vetor cada. Pré-calculas os vetores dos docs e indexas. Rápido, escala a milhões.

é o que vai para o vector store · retrieval

Cross-encoder

mete query e documento juntos no modelo e devolve uma pontuação de par. Muito mais preciso — e não indexável.

precisolentorerank do top-k

Padrão real: bi-encoder recupera os top-50 (barato), cross-encoder reordena esses 50 (caro mas só 50). O melhor dos dois.

71
Esta distinção é decisiva para quem constrói RAG. O bi-encoder codifica a pergunta e cada documento separadamente — duas passagens independentes pelo modelo, um vetor cada. A grande vantagem: os vetores dos documentos pré-calculam-se e indexam-se, então no momento da query só codificas a pergunta e fazes busca vetorial. Escala a milhões. A desvantagem: como nunca viu a pergunta e o documento ao mesmo tempo, perde interações finas. O cross-encoder faz o oposto: mete os dois juntos no modelo e produz uma pontuação de relevância do par. Muito mais preciso, mas não dá um vetor reutilizável — terias de correr o modelo para cada par query-documento, impossível a milhões. Por isso o padrão de produção é em duas fases: o bi-encoder recupera um top-50 barato, o cross-encoder reordena só esses 50. Recall barato, precisão onde importa.
Dimensionalidade

Mais dimensões: mais nuance, mais custo

384 leve · índice barato · menos nuance
768 o equilíbrio mais comum
1536 / 3072 mais nuance · storage e índice HNSW caros

A dimensão multiplica tudo: storage do vector store, memória do índice, custo de cada comparação. A milhões de vetores, escolher 1536 quando 384 chegava é dinheiro a arder — decisão de arquitetura.

72
As dimensões típicas: 384, 768, 1024, 1536, e os grandes vão a 3072. Mais dimensões dão ao modelo mais eixos para codificar nuance, e em geral mais qualidade de retrieval. Mas a dimensão multiplica todos os custos: o storage do vector store é linear na dimensão, a memória do índice HNSW também, e cada cálculo de semelhança percorre o vetor inteiro. A milhões de vetores, a diferença entre 384 e 1536 é quatro vezes o storage e o custo de índice — pode ser a diferença entre caber na RAM ou não. Por isso isto é uma decisão de arquitetura, não um default. A pergunta certa é: a tua tarefa precisa mesmo da nuance extra, ou um modelo de 384 dimensões recupera igualmente bem no teu corpus? E isto liga ao próximo conceito, que parte este trade-off pela raiz: e se pudesses escolher a dimensão depois de treinar?
Matryoshka (MRL)

Um vetor que se pode encurtar

Treinado para que os primeiros valores carreguem o essencial. Cortas o vetor em 256 e ainda funciona — uma só geração, várias dimensões. Indexas curto para triagem, guardas longo para rerank.

73
Matryoshka Representation Learning, MRL, resolve o trade-off da dimensão de forma elegante — daí o nome das bonecas russas. O modelo é treinado com uma perda que obriga os primeiros 256 valores do vetor a já serem, sozinhos, um bom embedding; os primeiros 768 a serem melhores; e o vetor completo o melhor de todos. A consequência prática é poderosa: geras o vetor uma vez, em dimensão máxima, mas podes truncá-lo — literalmente cortar e renormalizar — para a dimensão que quiseres, sem re-embeber. Isto permite um padrão muito eficiente: indexas a versão curta de 256 para fazer triagem rápida e barata sobre milhões, e guardas a versão longa para reordenar com precisão só o top que sobrou. Os modelos de embedding mais recentes da OpenAI e vários open-source suportam isto. É a resposta de engenharia ao slide anterior: já não tens de escolher a dimensão antes de saber se precisas dela.

O português tem armadilhas próprias. 77
Quatro métricas, mas uma manda. Recall@k: dos documentos relevantes, quantos apanhaste no top-k? Para RAG é a métrica-rei, porque o que não é recuperado simplesmente não chega ao LLM — ele não pode responder com o que não vê. MRR, mean reciprocal rank: olha para a posição do primeiro acerto, recompensa pô-lo em primeiro e penaliza pô-lo em nono. nDCG: o mais sofisticado, pondera por relevância graduada e pela posição, para quando "relevante" não é binário. Precision@k: a fração do top-k que é mesmo relevante — importa quando enches o contexto do LLM com ruído. A estratégia prática para RAG: foca-te primeiro no recall@k, sobe o k até recuperares quase sempre o documento certo, e só depois trazes um cross-encoder para reordenar e melhorar a precisão no topo. A ordem importa: recuperar mal é falhar antes de o gerador sequer começar.

RAG · chunking

O chunk é a unidade de tudo

O documento é grande demais para embeber inteiro e grande demais para caber no prompt. Então parte-se em pedaços — chunks.

Cada chunk vira um vetor, é a unidade que se recupera e a unidade que o LLM lê. Se o pedaço estiver mal cortado, todas as três sofrem.

A decisão mais barata e mais ignorada

Não há treino, não há GPU. É uma escolha de como partir texto. E decide se o RAG acerta ou falha.

84
Fixar a ideia antes das estratégias: o chunk é simultaneamente a unidade de embedding, de retrieval e de contexto. Um chunk demasiado grande dilui o vetor (já vimos na pooling — "a média dilui") e enche o prompt de ruído. Demasiado pequeno parte factos e perde contexto. O chunking é onde o engenheiro de dados tem mais controlo e menos código — e é por isso que é tantas vezes feito à toa (split a cada 1000 caracteres e segue).
Chunking · estratégia 1/5

Tamanho fixo — o corte cego

Corta a cada N caracteres ou tokens, ignorando o sentido. Barato e determinístico — e parte palavras, frases e factos a meio.

85
A baseline. Vantagem real: simples, previsível, idempotente — partes sempre igual. Defeito fatal: não sabe nada sobre a estrutura do texto, por isso o corte cai onde calha. É o ponto de partida de qualquer biblioteca (LangChain CharacterTextSplitter) e o culpado nº1 de respostas truncadas. Guardar o exemplo do facto partido para daqui a uns slides — aqui basta plantar a semente do problema.
Chunking · estratégia 2/5

Com overlap — a janela que desliza

Cada chunk reaproveita o fim do anterior (ex.: 200/50). O custo: mais chunks, mais vetores, índice maior. O ganho: factos de fronteira sobrevivem.

86
O patch mais comum ao corte fixo: sobrepor as janelas para que o que cai na fronteira de uma apareça inteiro na seguinte. Trade-off explícito de engenharia de dados: overlap de 25% significa ~25% mais vetores para armazenar e indexar, mais custo de embedding e índice maior. É a primeira vez na aula que se vê o custo de storage a depender de uma decisão de chunking — ligar de volta às "consequências de engenharia" dos embeddings.
Chunking · estratégia 3/5

Estrutural — cortar onde o documento já corta

Parágrafo, secção, título (Markdown ##, HTML, JSON). Os chunks ficam coerentes por construção — exige documentos com estrutura limpa.

87
Em vez de impor um tamanho, segue a estrutura que o autor já deu: parágrafos, headings, células. Funciona muito bem em FAQs, documentação técnica e Markdown — onde os limites lógicos já estão marcados. Cai mal em texto corrido sem estrutura (e aí o overlap volta a ser melhor). Mensagem: a melhor estratégia depende do corpus, não há default universal — adiantar o slide de trade-offs.
Chunking · estratégia 4/5

Semântico — cortar onde o tema muda

Embebe frase a frase e corta onde a similaridade cai. Chunks tematicamente puros — ao preço de embeber para depois chunkar.

88
O passo seguinte: deixar o conteúdo decidir o corte. Embebe-se cada frase, mede-se a similaridade entre vizinhas e corta-se nos vales — onde o assunto muda. Produz os chunks mais coerentes mas é o mais caro a construir (embeddings extra só para decidir os cortes) e não-determinístico se o modelo de embedding mudar. Bom exemplo de "mais inteligência no offline custa mais compute" — uma decisão de custo, não de IA.
Chunking · estratégia 5/5

Lê o layout real — tabelas inteiras, listas inteiras, cada título colado à sua secção. Essencial em PDFs, relatórios e documentos clínicos.

RAG · bases de dados vetoriais

Comparar com tudo não escala

Recuperar = encontrar os vetores mais próximos da pergunta. A forma exata é força bruta: cosseno contra todos.

Com 10M vetores, por pergunta, em milissegundos? Impossível. Daí a base vetorial: troca exatidão por velocidade.

ANN — Approximate Nearest Neighbour

Não devolve os k exatos, devolve os k quase certos, ordens de magnitude mais depressa. Um índice especializado para isto.

94
Motivar o índice antes de o desenhar. Busca exata = O(N) por query: calcular cosseno contra cada vetor. Aceitável para milhares, inviável para milhões com latência de serviço. A solução é Approximate Nearest Neighbour: aceitar uma resposta quase-exata (recall < 100%) em troca de velocidade brutal. É um trade-off de engenharia clássico — exatidão vs latência — e explica porque existem bases de dados vetoriais dedicadas em vez de um simples SELECT.
Índices · HNSW

HNSW — um mapa com atalhos

Entra na camada esparsa (saltos longos), desce greedy para a densa. Logarítmico em vez de linear — o índice por omissão em quase todas as bases vetoriais.

95
HNSW = Hierarchical Navigable Small World. Intuição sem matemática: várias camadas de grafo empilhadas. No topo, poucos nós com ligações longas — saltas grandes distâncias depressa. Cada camada abaixo é mais densa. A busca entra no topo, anda greedy para o vizinho mais próximo da pergunta, e quando não melhora desce uma camada e refina. Resultado: tempo ~logarítmico em vez de linear. Parâmetros que o engenheiro afina: M (ligações por nó) e ef (largura da busca) — mais recall custa mais memória/latência. Não precisam de implementar; precisam de saber que existe e o que se afina.
Índices · IVF

IVF — partir o espaço em bairros

Agrupa os vetores em clusters (centróides). A pergunta só compara com os poucos bairros mais próximos — nprobe regula quantos.

96
IVF = Inverted File Index. Alternativa ao HNSW: em vez de um grafo, parte-se o espaço em k clusters (k-means → centróides). Indexar = atribuir cada vetor ao seu cluster. Buscar = encontrar os centróides mais próximos da pergunta e só varrer esses clusters (nprobe). Trade-off: nprobe baixo = rápido mas pode falhar o vizinho se ele estiver num cluster vizinho; nprobe alto = mais recall, mais lento. HNSW costuma ganhar em recall/latência; IVF é mais leve em memória e bom para datasets enormes (e combina-se com quantização — IVF-PQ). O ponto pedagógico: dois modos de aproximar, ambos trocam exatidão por velocidade, ambos têm um botão de recall vs custo.

RAG · retrieval

Top-k — os k vizinhos mais próximos

A pergunta vira vetor pelo mesmo modelo dos chunks. Devolve os k de maior cosseno. Escolher k é trade-off: poucos = perdes contexto, muitos = enches de ruído.

98
O passo online mais simples e o mais fácil de fazer mal. Detalhe crítico de engenharia: a pergunta tem de ser embebida com EXATAMENTE o mesmo modelo (e versão) dos chunks — espaços diferentes não se comparam. O cosseno mede o ângulo, não a magnitude. k é um botão: k pequeno arrisca não trazer a passagem certa; k grande enche o contexto de quase-relevantes que distraem o LLM e custam tokens. Ligar ao chunking: top-k bom precisa de chunks bons por baixo.
RAG · busca híbrida

Vetor encontra sentido; BM25 encontra a palavra exata

Dois recalls que falham em sítios opostos. O vetor perde o código SKU-4471; o BM25 perde "enfarte" quando pediste "ataque cardíaco". Juntos cobrem-se.

99
Por que híbrida: a busca densa (vetor) é ótima para sentido e sinónimos mas falha em correspondências literais — números de série, siglas, nomes próprios, códigos raros que o embedding "suaviza". A busca esparsa (BM25, keyword clássica, o motor de um Elasticsearch) é o oposto: literal e cega à semântica. Correm-se as duas em paralelo e fundem-se os rankings, tipicamente com Reciprocal Rank Fusion (RRF) — soma de 1/(k+rank), sem precisar de calibrar escalas. Em produção, híbrido bate quase sempre qualquer dos dois sozinho. É infraestrutura de recuperação, não modelação.
RAG · reranking

Cross-encoder — ler a pergunta e o documento juntos

Recupera-se barato ~50 candidatos com o bi-encoder; reordenam-se com o cross-encoder, que lê o par em conjunto. Caro demais para todos — perfeito para os finalistas.

100
Reranking fecha o funil de retrieval. Distinção-chave: o bi-encoder (o que gera embeddings) codifica pergunta e documento separadamente, em vetores que se comparam por cosseno — barato porque os vetores dos chunks são pré-calculados, mas perde interação fina entre pergunta e texto. O cross-encoder mete pergunta e documento juntos na entrada do modelo e devolve um score de relevância — muito mais preciso porque vê a interação, mas caro porque corre um forward por par e nada se pré-calcula. Por isso a arquitetura em dois estágios: bi-encoder traz 50 candidatos rápido, cross-encoder reordena para o top-5 que vai ao LLM. Ganho de qualidade grande por pouco custo, porque só corre nos finalistas.
RAG · avaliação

Medir, não achar

Recall@k

O chunk certo veio no top-k? Mede o retrieval — falha aqui e o resto não salva.

Context precision

Dos k recuperados, quantos eram mesmo relevantes? Ruído distrai o LLM e gasta tokens.

Faithfulness

A resposta assenta no contexto recuperado, ou o modelo inventou? Groundedness.

Separar onde falha

Recall mau → problema de dados (chunking, embeddings, índice). Faithfulness má com bom contexto → problema de geração (prompt, modelo). Diagnósticos diferentes.

101
Sem avaliação, otimizar RAG é adivinhar. Três famílias: (1) métricas de retrieval — recall@k (a passagem certa está no top-k?) e context precision (proporção de recuperados que são relevantes); (2) métricas de geração — faithfulness/groundedness (a resposta é suportada pelo contexto, sem inventar) e answer relevance. O valor pedagógico maior: estas métricas localizam a falha. Se o recall é baixo, mexer no prompt não adianta — o problema é dados. Se o contexto está bom mas a resposta diverge, é geração. Ferramentas: RAGAS, etc. Isto transforma "o RAG está mau" numa hipótese testável.

102
Fecho da secção RAG e prova final da tese-zeitgeist. Em produção, os RAGs raramente falham por o LLM ser fraco; falham porque a plataforma de dados por baixo não tem disciplina: índices stale (já vimos, CDC resolve), schema drift que partiu a ingestão sem ninguém ver, backfills que misturaram vetores de modelos diferentes no mesmo índice (espaços incompatíveis — ligar a "trocar de modelo = re-embeber tudo"), e a falta de contratos de dados a montante. A boa notícia para um engenheiro de dados: o ferramental que já domina — observabilidade, data contracts, testes, lineage — é exatamente o que falta a estes sistemas. Esta é a oportunidade da Porta B.

A colisão de fundo da aula toda, agora explícita. A engenharia de dados foi construída sobre reprodutibilidade: reprocessar um job tem de dar o mesmo. O LLM é probabilístico por construção — mesmo a temperatura 0 não é garantia (hardware, batching, versões do modelo introduzem variação). As mitigações — schema-constrain, cache de saídas, validação, congelar versão do modelo — são justamente o que vimos no LLM-como-ETL. Não se elimina o estocástico; cerca-se.
Fio crítico · 2

Corrupção silenciosa

Não é um crash. É um número errado — ou um facto stale — que propaga a jusante durante semanas.

sem stack trace · sem alerta · descobre-se quando já decidiu algo

Porque é pior do que um bug

Um bug rebenta e tu vês. Isto corre verde. O SQL que mente, o chunk truncado, o vetor rançoso — todos falham em silêncio. A defesa é validação e testes, não esperança.

104
Costurar os fios da aula: o SQL sintaticamente perfeito que dá o número errado (Porta A), o chunk cortado a meio, o índice stale (RAG). Têm todos a mesma assinatura: não há exceção, não há vermelho, o pipeline reporta sucesso. O custo é diferido e composto — a decisão errada já foi tomada a montante quando o erro aparece. A única defesa é tratar as saídas de LLM como dados não-confiáveis: validar, testar contra expected, monitorizar distribuições, flag de baixa confiança. Repetir a regra da casa: avisar a azeitona, nunca o vermelho — mas o conteúdo é mesmo este alarme.
Fio crítico · 3

Custo e latência: um LLM sobre milhões de linhas

Um LLM por linha em 10M linhas é horas e uma fatura absurda. A engenharia está em não o chamar quando não é preciso — pré-filtrar, cachear, processar só o delta.

105
Ordem de grandeza concreta: uma chamada de LLM é dezenas a centenas de milissegundos e custa tokens; multiplica por 10 milhões e tens horas de wall-clock e uma fatura de quatro a cinco dígitos. Uma transformação SQL/regex sobre as mesmas linhas é segundos e custa praticamente nada. A maturidade de engenharia não é "LLM em tudo", é arquitetar para minimizar chamadas: pré-filtrar com regras o que é trivial, processar só o que mudou (CDC), cachear saídas determinísticas, agrupar em batch, e reservar o LLM para o subconjunto que mesmo precisa de compreensão. O LLM é um recurso caro de latência alta — trata-se como tal.
Fio crítico · 4

Quando NÃO usar um LLM

Uma regra chega

Formato fixo, parsing de logs, validação de tipos, joins, agregações. Determinístico, barato, auditável. Não convides o caos.

O LLM ganha

Onde as regras não chegam: texto livre, ambiguidade, variação infinita de forma, sentido que precisa de contexto.

negaçãonormalizaçãointençãolinguagem

A pergunta certa não é "consigo usar um LLM aqui?" — é "uma regra resolve isto mais barato e sem o risco?" Se sim, usa a regra.

106
O contrapeso honesto a uma aula que passou muito tempo a entusiasmar com a Porta B. O reflexo do hype é meter LLM em tudo; o reflexo de engenheiro é perguntar se uma regra determinística resolve — porque a regra é mais barata, mais rápida, reprodutível e auditável. Voltar ao caso dos logs do slide 14: ali o regex bate o LLM em todas as dimensões. O LLM justifica-se exatamente onde as regras explodem em complexidade: negação, normalização de sinónimos, ambiguidade, intenção, variação linguística sem fim. Critério de decisão, não ideologia. Isto é o "olho crítico" do arco da aula.

A disciplina que isto premeia é a mesma que já era boa prática — só que agora paga mais.
